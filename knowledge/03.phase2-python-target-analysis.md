# Phase2 Python 전환 대상 분석

이 문서는 전자정부프레임워크 3.x -> 4.3 전환 과정에서, OpenRewrite만으로 처리하기 어려운 항목을 Python 프로그램 관점에서 분류한 TO-BE 후보 조사 결과다.

목적은 아래 두 가지다.

* 어떤 항목을 Phase2 Python Rule Engine이 맡아야 하는지 정리한다.
* 어떤 항목은 자동 변환, 조건부 자동 변환, 수동 검토로 나눠야 하는지 판단 기준을 남긴다.

---

## 1. Phase2가 필요한 이유

OpenRewrite는 `pom.xml`, Java import/type, XML 문자열 치환처럼 전후 매핑이 명확한 작업에는 강하다. 반면 아래 영역은 단순 치환만으로 끝나지 않는다.

* DAO 상속 구조와 공통 래퍼 구조 분석
* iBatis SQL Map 문법을 MyBatis XML 문법으로 구조 변환
* Spring XML에서 `sqlMapClient` 연계 설정을 `SqlSessionFactory` 기준으로 재구성
* 변환 후 잔존 패턴 탐지와 수동 검토 대상 분류

즉, Phase2는 "치환"보다 "구조 해석 + 규칙 적용 + 후처리 검증"이 필요한 영역이다.

## 2. Python 전환 대상 분류

## 2-1. 1순위: Python 자동 변환 핵심 대상

### DAO 구조 변환

주요 대상:

* `EgovAbstractDAO` 상속 클래스
* 프로젝트 공통 DAO 래퍼
* `list`, `select`, `insert`, `update`, `delete` 호출 패턴
* `SqlMapClientTemplate` 직접 사용 코드

대표 예:

```java
extends EgovAbstractDAO
```

→

```java
extends EgovAbstractMapper
```

단, 단순 상속 변경만으로 끝나지 않는다.

* 공통 래퍼가 `EgovAbstractDAO`를 감싸고 있는지 확인 필요
* 호출 메서드가 MyBatis 기준 메서드와 1:1 대응하는지 확인 필요
* 리턴 타입과 캐스팅 구조까지 확인 필요

### SQL Map -> MyBatis XML 변환

주요 대상:

* `#var#`, `$var$`
* `<sqlMap>` -> `<mapper>`
* `parameterClass`, `resultClass`
* `<dynamic>`, `<isNotEmpty>`, `<isEqual>`, `<iterate>`

대표 예:

```xml
#id#
```

→

```xml
#{id}
```

이 영역은 단순 텍스트 치환이 아니라 XML 구조를 해석한 뒤 규칙을 적용해야 하므로 Python 변환 대상에 포함한다.

### Spring XML의 iBatis 연계 설정 변환

주요 대상:

* `org.springframework.orm.ibatis.SqlMapClientFactoryBean`
* `SqlMapClientTemplate`
* `configLocation`
* `mapperLocations`
* DAO / mapper / Spring bean 참조 구조

대표 예:

```xml
org.springframework.orm.ibatis.SqlMapClientFactoryBean
```

→

```xml
org.mybatis.spring.SqlSessionFactoryBean
```

단순 class 이름 변경만으로 끝나지 않고, bean id, property, mapper 연결 구조까지 함께 정리해야 한다.

## 2-2. 2순위: 조건부 자동 변환 대상

아래 항목은 Python으로 일부 자동화할 수 있으나, 프로젝트 편차가 커서 조건부로 접근하는 것이 안전하다.

### sqlMapConfig -> mapperLocations 확장

주요 이슈:

* DB별 sql-map-config 파일 분리 여부
* mapper xml 경로 패턴 차이
* Spring bean에서 참조하는 방식 차이

조건:

* 경로 패턴이 명확한 경우 자동 확장
* 경로 계산이 불명확하면 수동 검토 경고만 남김

### 공통 DAO 래퍼 연쇄 변환

주요 이슈:

* 프로젝트 내부 `EgovComAbstractDAO` 같은 중간 래퍼 존재
* 래퍼 내부의 `getSqlMapClientTemplate()` 호출 여부
* 하위 DAO가 직접 mapper namespace를 노출하지 않는 구조

조건:

* 공통 패턴이면 자동 변환
* 프로젝트별 커스텀 API가 섞이면 수동 검토로 분류

### 동적 SQL 일부 변환

주요 이슈:

* `<iterate>`의 separator/open/close 조합
* 중첩 `<dynamic>` 구조
* `$var$` 사용 위치가 바인딩이 아니라 SQL 조각일 가능성

조건:

* 명확한 패턴은 자동 변환
* SQL 의미 해석이 필요하면 경고만 남김

## 2-3. 후순위: 수동 검토 또는 Phase3 연계 대상

아래 항목은 Python 규칙만으로 안전하게 끝내기 어렵다.

### Validation

대상:

* `spring-modules-validation`
* `DefaultBeanValidator`
* `validator-rules.xml`
* `validation.xml`
* VO annotation 전환

판단:

* Java, XML, 설정 파일이 함께 얽혀 있어 별도 트랙으로 관리

### Quartz

대상:

* Quartz 1.x -> 2.x
* job / trigger 설정
* 스케줄 등록 방식

판단:

* API 차이와 운영 영향 검토가 필요하므로 개별 검토 대상

### 솔루션 연계 / 특수 인프라 설정

예:

* OZ
* Petra
* Xecure
* JCAOS

판단:

* 전환 공통 규칙으로 일반화하기 어려워 수동 검토 또는 Phase3 보정 대상

## 3. 자동화 가능 수준 분류 기준

### 자동 변환 가능

아래 조건을 만족하면 Python 자동 변환 대상으로 본다.

* 전후 구조 매핑이 명확하다
* 소스에서 패턴 식별이 안정적이다
* 변환 후 기계적으로 검증 가능하다

예:

* `EgovAbstractDAO` 상속 클래스
* `#var#` -> `#{var}`
* `SqlMapClientFactoryBean` -> `SqlSessionFactoryBean`

### 조건부 자동 변환

아래 조건이면 자동 변환 후 경고를 남긴다.

* 대부분의 패턴은 공통이지만 일부 예외가 존재한다
* 프로젝트 편차가 커서 참조 구조 확인이 필요하다

예:

* 공통 DAO 래퍼
* `sqlMapConfig` 기반 mapper 경로 계산
* 중간 수준의 동적 SQL

### 수동 검토 필요

아래 조건이면 수동 검토 또는 Phase3 대상으로 넘긴다.

* SQL 의미 해석이 필요하다
* 업무 로직 판단이 개입된다
* 외부 솔루션이나 특수 설정이 연결되어 있다

예:

* 복잡한 `<iterate>` / 중첩 `<dynamic>`
* 커스텀 DAO 헬퍼
* Validator / Quartz / 솔루션 연계

## 4. Phase2 프로그램 요구사항으로 연결되는 항목

이 분석 결과는 Phase2 프로그램 설계에 아래 요구사항으로 연결된다.

* `discover`는 DAO, SQL Map, Spring XML의 위치와 종류를 전수 수집해야 한다
* `classify`는 iBatis only / mixed persistence / DAO wrapper 존재 여부를 판단해야 한다
* `transform_*`는 구조 변경과 문자열 치환을 함께 수행해야 한다
* `postcheck`는 잔존 iBatis 패턴과 수동 검토 대상을 탐지해야 한다
* `report`는 자동 변환 범위와 검토 필요 범위를 명확히 분리해서 보여줘야 한다

---

## 5. 결론

Phase2 Python Rule Engine은 아래 세 영역을 핵심 범위로 가져가는 것이 적절하다.

* DAO 구조 변환
* SQL Map -> MyBatis XML 변환
* Spring XML의 iBatis 연계 설정 변환

그리고 아래 항목은 별도 관리가 필요하다.

* 조건부 자동 변환: 공통 DAO 래퍼, `sqlMapConfig`, 중간 난도의 동적 SQL
* 수동 검토 또는 Phase3 연계: Validation, Quartz, 솔루션 연계, 복잡한 업무 SQL
