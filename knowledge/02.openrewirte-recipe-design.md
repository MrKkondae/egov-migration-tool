# OpenRewrite Recipe 구성 방안

1순위 대상 라이브러리 분석 결과, OpenRewrite는 상위 레벨에서 `pom`, `java`, `xml` 3개 영역으로 분리하여 관리하고, `pom` 영역은 다시 성격별 하위 recipe로 나누어 관리하는 것을 원칙으로 한다.

```text
rules/
└── phase1-openrewrite/
    ├── 01.pom-dependency-migration.yml
    ├── 02.java-migration.yml
    ├── 03.xml-migration.yml
    ├── pom/
    │   ├── egovframe-coordinates.yml
    │   ├── spring-upgrades.yml
    │   ├── logging-upgrades.yml
    │   ├── jackson-jdbc-jstl.yml
    │   └── safe-common-version-upgrades.yml
    ├── java/
    │   ├── egov-package-renames.yml
    │   └── jackson-type-migration.yml
    └── xml/
        ├── egov-xml-package-renames.yml
        └── jackson-xml-migration.yml
```

## 1. 01.pom-dependency-migration.yml

### 목적

`pom.xml` 변환용 composite recipe의 진입점으로 사용한다.

### 역할

이 파일은 개별 의존성 규칙을 직접 길게 작성하지 않고, `pom/` 하위의 세부 recipe를 순서대로 호출하는 recipe만 둔다.

### recipeList 구성

```yaml
recipeList:
  - egov.migration.phase1.pom.EgovframeCoordinates
  - egov.migration.phase1.pom.SpringUpgrades
  - egov.migration.phase1.pom.LoggingUpgrades
  - egov.migration.phase1.pom.JacksonJdbcJstl
  - egov.migration.phase1.pom.SafeCommonVersionUpgrades
```

### 특징

* `mvn rewrite:dryRun` 실행 시 전체 `pom.xml` 변환 진입점으로 사용 가능
* 개별 recipe를 선택적으로 활성화하기 쉽다
* 문제가 발생해도 어느 묶음에서 발생했는지 추적이 쉽다

---

## 2. pom/ 하위 recipe 구성

`pom` 영역은 성격이 다른 변경을 한 파일에 모두 넣지 않고, 위험도와 검증 관점을 기준으로 5개 recipe로 분리한다.

### 2-1. egovframe-coordinates.yml

#### 목적

eGovFrame 3.1 계열 RTE 의존성을 eGovFrame 4.3 좌표로 1:1 치환한다.

#### 적용 대상

* `egovframework.rte.fdl.cmmn` → `org.egovframe.rte.fdl.cmmn`
* `egovframework.rte.fdl.idgnr` → `org.egovframe.rte.fdl.idgnr`
* `egovframework.rte.fdl.logging` → `org.egovframe.rte.fdl.logging`
* `egovframework.rte.fdl.property` → `org.egovframe.rte.fdl.property`
* `egovframework.rte.psl.dataaccess` → `org.egovframe.rte.psl.dataaccess`
* `egovframework.rte.ptl.mvc` → `org.egovframe.rte.ptl.mvc`

#### 적용 방식

* `org.openrewrite.maven.ChangeDependencyGroupIdAndArtifactId`
* `changeManagedDependency: true`

#### 특징

* 가장 안정적인 1차 자동 변환 영역
* direct dependency와 `dependencyManagement`를 함께 고려 가능

### 2-2. spring-upgrades.yml

#### 목적

Spring 3.2.9.RELEASE 계열 의존성을 5.3.37 기준으로 정리한다.

#### 적용 대상

* `spring-aop`
* `spring-beans`
* `spring-context`
* `spring-context-support`
* `spring-core`
* `spring-expression`
* `spring-jdbc`
* `spring-orm`
* `spring-tx`
* `spring-web`
* `spring-webmvc`

#### 적용 방식

* `org.openrewrite.maven.UpgradeDependencyVersion`
* `overrideManagedVersion: false`

#### 특징

* Spring 계열 버전 정합성을 한 묶음으로 검토 가능
* parent/BOM 관리 버전을 강제로 깨지 않도록 보수적으로 구성

### 2-3. logging-upgrades.yml

#### 목적

SLF4J / Log4j2 계열 버전을 정리하여 보안 및 버전 정합성을 맞춘다.

#### 적용 대상

* `slf4j-api`
* `jcl-over-slf4j`
* `log4j-over-slf4j`
* `log4j-api`
* `log4j-core`
* `log4j-slf4j-impl`

#### 적용 방식

* `org.openrewrite.maven.UpgradeDependencyVersion`
* `overrideManagedVersion: false`

#### 특징

* 버전 통일만 수행하고 bridge 추가/제거는 자동화하지 않음
* 로깅 구조 충돌 여부는 별도 dry-run 및 classpath 검토 필요

### 2-4. jackson-jdbc-jstl.yml

#### 목적

즉시 적용 가능한 비-Spring 계열 `pom.xml` 치환 규칙들을 한 묶음으로 관리한다.

#### 적용 대상

* Jackson 1.x → 2.x
* MyBatis / MyBatis-Spring 버전업
* OJDBC `ojdbc14` → `ojdbc8`
* JSTL 구현체 치환

#### 세부 구성

##### Jackson

* `jackson-core-asl` → `jackson-core`
* `jackson-mapper-asl` → `jackson-databind`

##### MyBatis

* `mybatis` → `3.5.16`
* `mybatis-spring` → `2.1.2`

##### OJDBC

* `com.oracle:ojdbc14` → `com.oracle.database.jdbc:ojdbc8`
* `com.oracle.database.jdbc:ojdbc14` → `com.oracle.database.jdbc:ojdbc8`

##### JSTL

* `taglibs:standard` → `org.apache.taglibs:taglibs-standard-impl`
* `javax.servlet:jstl` → `1.2`

#### 특징

* 좌표 치환과 버전업이 섞여 있으나 모두 `pom.xml` 수준에서 보수적으로 적용 가능
* `jdbc-14.jar` 같은 local/systemPath 선언은 자동화 대상에서 제외
* `jackson-annotations` 같은 신규 의존성은 무조건 추가하지 않음

### 2-5. safe-common-version-upgrades.yml

#### 목적

상대적으로 안전한 common utility 계열 버전업을 별도 recipe로 관리한다.

#### 적용 대상

* `commons-beanutils`
* `commons-io`
* `commons-lang3`
* `commons-net`

#### 적용 방식

* `org.openrewrite.maven.UpgradeDependencyVersion`
* `overrideManagedVersion: false`

#### 특징

* 필요 시 가볍게 on/off 할 수 있는 보수적 묶음
* 실제 소스 영향은 적지만 회귀 테스트는 필요

## 3. java/ 영역

### 목적

Java 소스의 import, package, type reference 변경 중에서 소스 의미 변화 없이 비교적 안전하게 자동화할 수 있는 보수적 치환만 수행한다.

### 현재 구성

`java` 영역은 `pom`과 동일하게 composite recipe + 하위 recipe 구조로 관리한다.

```text
rules/
└── phase1-openrewrite/
    ├── 02.java-migration.yml
    └── java/
        ├── egov-package-renames.yml
        └── jackson-type-migration.yml
```

### 3-1. 02.java-migration.yml

#### 목적

Java 소스 변환용 composite recipe 진입점으로 사용한다.

#### recipeList 구성

```yaml
recipeList:
  - egov.migration.phase1.java.EgovPackageRenames
  - egov.migration.phase1.java.JacksonTypeMigration
```

#### 특징

* 현재는 1차 자동변환 범위만 포함한다
* 위험도가 높은 Java 규칙은 주석으로만 남기고 실제 recipe에는 넣지 않는다

### 3-2. egov-package-renames.yml

#### 목적

eGovFrame 3.1 Java 소스의 `egovframework.rte.*` 패키지를 `org.egovframe.rte.*`로 일괄 변경한다.

#### 적용 방식

* `org.openrewrite.java.ChangePackage`
* `recursive: true`

#### 적용 예시

```java
import egovframework.rte.psl.dataaccess.EgovAbstractMapper;
```

→

```java
import org.egovframe.rte.psl.dataaccess.EgovAbstractMapper;
```

#### 특징

* eGov Runtime 패키지 변경에 가장 직접적으로 대응하는 Java recipe
* method, DAO 구조, 비즈니스 로직은 건드리지 않음

### 3-3. jackson-type-migration.yml

#### 목적

Jackson 1.x 기반 Java 타입과 annotation package를 Jackson 2.x 기준으로 변경한다.

#### 적용 방식

* `org.openrewrite.java.ChangePackage`
* `org.openrewrite.java.ChangeType`

#### 적용 예시

```java
import org.codehaus.jackson.map.ObjectMapper;
```

→

```java
import com.fasterxml.jackson.databind.ObjectMapper;
```

#### 포함 범위

* `org.codehaus.jackson.annotate.*` → `com.fasterxml.jackson.annotation.*`
* `org.codehaus.jackson.map.annotate.*` → `com.fasterxml.jackson.databind.annotation.*`
* `ObjectMapper`
* `JsonNode`
* `JsonFactory`
* `JsonParser`
* `JsonGenerator`
* `JsonToken`
* `TypeReference`
* `JsonIgnoreProperties`
* `MappingJacksonHttpMessageConverter` → `MappingJackson2HttpMessageConverter`

### 보류 권장 항목

현재 `java` YAML에는 아래 항목을 실제 규칙으로 넣지 않고, composite recipe 주석에만 보류 후보로 남긴다.

* Commons DBCP type 변경
  `org.apache.commons.dbcp.BasicDataSource` → `org.apache.commons.dbcp2.BasicDataSource`
* `EgovAbstractDAO` → `EgovAbstractMapper`
* `org.apache.log4j.Logger` → `org.slf4j.Logger`
* `org.jdom.*` → `org.jdom2.*`
* 메서드명 치환 및 invocation rewrite

### 보류 사유

* Java import/type 변경만으로 끝나지 않고 XML, 설정, API 구조 변경이 함께 필요한 경우가 있음
* 실제 소스가 없거나 닫힌망 환경이라 회귀 여부를 즉시 검증할 수 없음
* 1차 자동변환 범위에서는 method-level rewrite를 제외하기로 결정함

### 특징

* Java 소스 자동 변환에 집중
* 현재는 package/type/import 치환 중심의 보수적 구성
* 컴파일 검증으로 결과 확인 가능

## 4. xml/ 영역

### 목적

XML 영역은 Spring Context XML, Spring MVC XML, sql-map XML 등에서 문자열 수준으로 안전하게 치환 가능한 항목만 phase1에서 자동 변환한다.

현재 XML 영역도 `pom`, `java`와 동일하게 composite recipe + 하위 recipe 구조로 관리한다.

```text
rules/
└── phase1-openrewrite/
    ├── 03.xml-migration.yml
    └── xml/
        ├── egov-xml-package-renames.yml
        └── jackson-xml-migration.yml
```

### 4-1. 03.xml-migration.yml

#### 목적

XML 변환용 composite recipe 진입점으로 사용한다.

#### recipeList 구성

```yaml
recipeList:
  - egov.migration.phase1.xml.EgovXmlPackageRenames
  - egov.migration.phase1.xml.JacksonXmlMigration
```

#### 특징

* phase1에서는 XML 구조 자체를 바꾸지 않고 문자열 치환 중심으로만 적용한다
* bean wiring, bean id, property 구조 변경이 필요한 항목은 포함하지 않는다

### 4-2. egov-xml-package-renames.yml

#### 목적

XML 내부의 `egovframework.rte.*` 문자열 참조를 `org.egovframe.rte.*`로 일괄 치환한다.

#### 적용 대상

* Spring bean `class` 속성
* exception mapping 등 XML 텍스트 값
* iBatis SQL Map 내부 `typeAlias` 같은 fully-qualified type 문자열

#### 적용 방식

* `org.openrewrite.text.FindAndReplace`
* `find: egovframework.rte`
* `replace: org.egovframe.rte`
* `filePattern: "**/*.xml"`

#### 적용 예시

```xml
<bean class="egovframework.rte.fdl.property.impl.EgovPropertyServiceImpl">
```

→

```xml
<bean class="org.egovframe.rte.fdl.property.impl.EgovPropertyServiceImpl">
```

### 4-3. jackson-xml-migration.yml

#### 목적

Spring MVC XML 설정에서 Jackson 1.x converter class name을 Jackson 2.x 표준 class name으로 치환한다.

#### 적용 방식

* `org.openrewrite.text.FindAndReplace`
* `find: MappingJacksonHttpMessageConverter`
* `replace: MappingJackson2HttpMessageConverter`
* `filePattern: "**/*.xml"`

### 대상 예시

#### Jackson Converter 변경

```xml
MappingJacksonHttpMessageConverter
```

→

```xml
MappingJackson2HttpMessageConverter
```

#### eGov package rename

```xml
egovframework.rte.psl.dataaccess.util.EgovMap
```

→

```xml
org.egovframe.rte.psl.dataaccess.util.EgovMap
```

### 보류 권장 항목

현재 `xml` YAML에는 아래 항목을 실제 규칙으로 넣지 않고, composite recipe 주석에만 보류 후보로 남긴다.

* `SqlMapClientFactoryBean` → `SqlSessionFactoryBean`
* `org.apache.commons.dbcp.BasicDataSource` → `org.apache.commons.dbcp2.BasicDataSource`
* `DefaultBeanValidator` 관련 bean 정의 제거 또는 Bean Validation 전환

### 보류 사유

* bean class 이름만 바꾸면 끝나지 않고 bean id, 참조 관계, property 구조가 함께 바뀔 수 있음
* iBatis → MyBatis 전환은 DAO, SQL Map XML, Spring bean 구성을 함께 다뤄야 함
* validator 전환은 XML뿐 아니라 Java annotation 및 검증 규칙 파일 정리가 필요함

### 특징

* phase1 XML recipe는 보수적인 text-level replacement만 수행
* `pom`, `java`보다 런타임 영향 가능성이 커서 적용 후 기동 테스트가 필요
* 구조 변경형 XML 전환은 phase2 또는 수동 검토 대상으로 분리

---

## 적용 순서

```text
1. pom/ 하위 recipe 개별 dry-run
   - egovframe-coordinates.yml
   - spring-upgrades.yml
   - logging-upgrades.yml
   - jackson-jdbc-jstl.yml
   - safe-common-version-upgrades.yml
2. 01.pom-dependency-migration.yml composite 실행
3. java/ 영역 recipe 적용
4. xml/ 영역 recipe 적용
5. Python Rule
6. LLM 보정
```

### 적용 원칙

* `pom` 영역은 먼저 하위 recipe 단위로 `mvn rewrite:dryRun -Drewrite.activeRecipes=...` 검토를 선행한다
* 개별 recipe 검토가 끝난 뒤 `01.pom-dependency-migration.yml`을 통해 전체 `pom.xml` 변환을 한 번에 실행한다
* `java` 영역은 `pom.xml` 변경 후 컴파일 오류를 기준으로 적용 범위를 조정한다
* `xml` 영역은 가장 마지막에 적용하고, 적용 후에는 기동 테스트까지 포함해 검증한다

현재 Library Matrix 기준으로 OpenRewrite 적용 범위는 다음과 같이 정의한다.

* `pom.xml` 의존성 좌표 변경
* `pom.xml` 의존성 버전업
* Java import 변경
* Java type 변경
* XML 설정 변경

복잡한 API 변경, SQL 변환, Framework 구조 변경, 업무 소스 전용 라이브러리 전환은 OpenRewrite 대상에서 제외하고 Python Rule 또는 LLM 보정 단계에서 처리한다.
