# Phase2 수동 검토 정책 초안

이 문서는 Phase2 변환 이후 어떤 항목을 수동 검토 대상으로 보고, 어떤 항목을 Phase3 또는 추가 규칙 보강으로 넘길지 판단하는 기준을 정리한 초안이다.

목적은 아래와 같다.

* `phase2-report.md`의 경고를 일관된 기준으로 해석한다.
* 자동 변환 완료와 전환 완료를 혼동하지 않도록 한다.
* 반복적으로 발생하는 예외를 규칙 환류 대상으로 관리한다.

---

## 1. 기본 원칙

### 자동 변환 성공과 최종 전환 완료는 다르다

파일이 수정되었더라도 아래가 남아 있으면 검토가 필요하다.

* iBatis 잔존 문법
* Spring bean 참조 불일치
* DAO 호출 의미 불명확
* 외부 솔루션/업무 SQL 영향

### 검토 기준은 파일 단위보다 이슈 단위가 우선이다

한 파일에 경고가 3개 있어도 하나의 원인에서 나온 경고라면 함께 검토한다.

### 반복되는 경고는 규칙 보강 후보로 본다

같은 유형의 경고가 여러 파일에서 반복되면 수동 처리로 끝내지 않고 Python 또는 OpenRewrite 규칙 보강 대상으로 올린다.

---

## 2. 수동 검토 대상 분류

## 2-1. 즉시 수동 검토 필요

아래 항목은 보고서에 나오면 우선 검토한다.

### 복잡한 동적 SQL

예:

* `<iterate>`
* 중첩 `<dynamic>`
* `$var$` 기반 SQL 조립

이유:

* SQL 의미가 바뀔 위험이 있다
* 자동 변환이 문법상 통과해도 실행 의미가 달라질 수 있다

### 공통 DAO 래퍼 / 커스텀 헬퍼

예:

* `EgovComAbstractDAO`
* 프로젝트 고유 DAO utility

이유:

* 공통 래퍼 내부 동작을 확인해야 하므로 파일 하나만 보고 판단하기 어렵다

### Spring XML 특수 설정

예:

* `lobHandler`
* 다중 datasource
* 수동 wiring이 많은 bean 설정

이유:

* bean class 이름만 바꿔서는 안전하지 않다

### 외부 솔루션 연계

예:

* OZ
* Petra
* Xecure
* JCAOS

이유:

* 제품별 설정과 런타임 제약이 달라 공통 규칙 적용이 어렵다

## 2-2. 조건부 수동 검토

아래 항목은 변환 결과와 잔존 패턴을 함께 보고 검토 우선순위를 정한다.

### `sqlMapConfig` 기반 mapper 경로

기준:

* `mapperLocations`가 명확히 계산되면 우선순위 낮음
* config 파일 분기와 경로 규칙이 복잡하면 우선순위 높음

### DAO API 치환 결과

기준:

* `selectOne`, `selectList` 등으로 자연스럽게 바뀌었는지 확인
* 캐스팅, 제네릭, 리턴 타입이 어색하면 검토

### 잔존 iBatis 패턴

기준:

* 단순 문자열 잔존인지
* 실제 실행 경로의 중요 SQL인지

## 3. Phase3 연계 기준

아래 경우는 Python 수동 수정만으로 끝내기보다 Phase3 보정 대상으로 넘기는 것이 효율적이다.

### 업무 의미 해석이 필요한 경우

예:

* 복잡한 SQL 조건식
* DAO 메서드 의미 재구성
* 서비스/컨트롤러까지 문맥을 봐야 하는 경우

### 코드 패턴이 프로젝트마다 다른 경우

예:

* 동일 기능인데 프로젝트별 helper 구조가 다른 경우
* 공통 규칙을 바로 만들기 어려운 경우

### 설명 가능한 보정이 필요한 경우

예:

* 왜 이 SQL을 `<if>` / `<foreach>`로 바꿨는지 근거가 필요한 경우
* 사람이 검토하기 쉽도록 변환 의도를 남겨야 하는 경우

---

## 4. 규칙 보강으로 환류할 항목

수동 검토에서 아래 조건을 만족하면 규칙 보강 후보로 올린다.

* 동일 경고가 3건 이상 반복됨
* 전후 매핑이 사실상 고정적임
* 기계적으로 검증 가능한 패턴임

대표 후보:

* 특정 형태의 DAO wrapper
* 반복되는 `parameterClass` / `resultClass` 변환 누락
* 동일 구조의 `sqlMapConfig` 경로 확장
* 특정 bean wiring 패턴

## 5. 보고서 해석 기준

`phase2-report.md`를 볼 때는 아래 순서로 확인한다.

1. `manual_review_file_count`
   전체 검토 규모를 파악한다.
2. `residual_warning_count`
   변환 후 남은 핵심 경고 개수를 본다.
3. 경고 유형 요약
   어떤 종류의 문제가 많은지 본다.
4. 주요 수동 검토 파일
   실제 검토 우선순위를 정한다.

특히 아래 경고는 우선순위를 높게 본다.

* `SQL Map Config`
* `DAO 메서드 호출 후속 확인`
* `공통 기반 클래스 상속 DAO`
* `Spring sqlMapClient 참조`

## 6. 검토 체크리스트

수동 검토 시 아래 체크리스트를 사용한다.

### DAO

* 상속 구조가 올바르게 바뀌었는가
* 호출 메서드가 의도한 MyBatis API로 바뀌었는가
* 공통 DAO 래퍼 영향이 없는가

### SQL Map / Mapper XML

* `#...#`가 `#{...}`로 적절히 바뀌었는가
* `${}` 사용이 안전한가
* 동적 SQL 의미가 유지되는가

### Spring XML

* `SqlSessionFactoryBean`으로 전환되었는가
* `mapperLocations`가 올바른가
* DAO / mapper / bean 참조가 연결되는가

### 빌드 / 실행

* 컴파일 오류가 없는가
* mapper namespace / statement id mismatch가 없는가
* 주요 기능 시나리오에서 SQL 실행이 정상인가

## 7. 결론

Phase2 수동 검토 정책의 핵심은 "자동 변환 결과를 빠르게 신뢰 구간과 위험 구간으로 나누는 것"이다.

따라서 보고서는 단순 변경 목록보다 아래를 잘 보여줘야 한다.

* 왜 검토가 필요한지
* 어떤 유형의 위험인지
* 규칙 보강 후보인지
* Phase3로 넘길지 여부
