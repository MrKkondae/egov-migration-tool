# Phase2 규칙 설계 초안

이 문서는 Phase2 Python Rule Engine의 규칙 설계 원칙과 주요 처리 흐름을 정리한 초안이다.

목적은 아래와 같다.

* `tools/conversion/` 프로그램이 어떤 판단 순서로 동작해야 하는지 정리한다.
* DAO, SQL Map, Spring XML 변환 규칙의 설계 기준을 문서화한다.
* 보고서와 후처리 경고가 어떤 기준으로 만들어져야 하는지 남긴다.

---

## 1. 설계 원칙

Phase2 규칙은 다음 원칙을 따른다.

### 1. 구조 해석 우선

문자열 치환 전에 파일 종류와 구조를 먼저 해석한다.

* Java는 상속, import, 메서드 호출 패턴을 본다
* XML은 루트 태그, 속성, namespace, 하위 태그 구조를 본다
* Spring XML은 bean/property 참조 관계를 본다

### 2. 안전한 자동화 우선

불확실한 변환은 억지로 자동 수정하지 않고 경고로 남긴다.

### 3. 변환과 검증을 분리하지 않음

변환 후 잔존 패턴 탐지와 수동 검토 대상 분류를 같은 파이프라인 안에서 처리한다.

### 4. Markdown과 JSON을 함께 생성

* JSON은 후속 프로그램 입력용
* Markdown은 사람 검토용

---

## 2. 전체 파이프라인 설계

`run_phase2.py`는 아래 흐름으로 동작한다.

```text
prepare_working_tree
    ↓
discover
    ↓
classify
    ↓
merge optional analysis json
    ↓
transform_sqlmap
    ↓
transform_dao
    ↓
transform_spring_xml
    ↓
postcheck
    ↓
report
```

각 단계의 역할은 아래와 같다.

### prepare_working_tree

* 원본을 작업 디렉터리로 복사하거나 작업 루트를 준비한다

### discover

* Java, SQL Map, Spring XML 위치를 수집한다
* 변환 대상 파일 수를 집계한다
* iBatis 관련 흔적을 전수 탐지한다

### classify

* iBatis only / MyBatis only / mixed persistence 분류
* DAO wrapper 존재 여부 확인
* 수동 검토 가능성이 높은 프로젝트인지 판단

### transform_sqlmap

* SQL Map 문법을 MyBatis XML 기준으로 변환한다
* 조건부 변환은 경고를 남긴다

### transform_dao

* DAO 상속 구조와 호출 패턴을 변환한다
* 공통 래퍼와 직접 사용 패턴을 함께 처리한다

### transform_spring_xml

* `SqlMapClient` 연계 설정을 `SqlSessionFactory` 기준으로 정리한다
* `mapperLocations`와 bean 참조를 정리한다

### postcheck

* 잔존 iBatis 패턴 탐지
* DAO / mapper / Spring 연결 불일치 탐지
* 수동 검토 대상 경고 수집

### report

* `phase2-discovery.json`
* `phase2-report.json`
* `phase2-report.md`

를 생성한다.

## 3. DAO 규칙 설계

### 핵심 규칙

#### 상속 구조 변경

```java
extends EgovAbstractDAO
```

→

```java
extends EgovAbstractMapper
```

#### import 정리

* `egovframework.rte.psl.dataaccess.EgovAbstractDAO`
* `egovframework.rte.psl.dataaccess.EgovAbstractMapper`
* `org.egovframe.rte.psl.dataaccess.EgovAbstractMapper`

위 조합을 현재 코드 상태에 맞게 정리해야 한다.

#### 메서드 호출 전환

대표 예:

* `list(...)` -> `selectList(...)`
* `queryForObject(...)` -> `selectOne(...)`
* `queryForList(...)` -> `selectList(...)`

단, 실제 치환 전 아래 확인이 필요하다.

* 리턴 타입이 단건인지 목록인지
* 캐스팅이 필요한지
* 공통 DAO 래퍼가 메서드를 감싸고 있는지

### DAO 경고 규칙

아래 경우 경고를 남긴다.

* 공통 DAO 래퍼 존재
* `SqlMapClientTemplate` 직접 사용
* 의미 해석이 필요한 커스텀 helper 호출
* 변환 후에도 `EgovAbstractDAO` 잔존

## 4. SQL Map 규칙 설계

### 핵심 규칙

#### 바인딩 문법 변환

```xml
#var#
```

→

```xml
#{var}
```

#### 루트 태그 변환

```xml
<sqlMap>
```

→

```xml
<mapper>
```

#### 속성명 변환

* `parameterClass` -> `parameterType`
* `resultClass` -> `resultType`

#### 동적 SQL 태그 변환

대상:

* `<dynamic>`
* `<isNotEmpty>`
* `<isEqual>`
* `<iterate>`

기본 방침:

* 기계적으로 변환 가능한 패턴만 자동 변환
* SQL 의미 해석이 필요한 경우 경고로 남김

### SQL Map 경고 규칙

아래 경우 경고를 남긴다.

* `${}` 또는 `$var$`가 SQL 조립에 사용됨
* 중첩 동적 태그가 복잡함
* namespace / statement id 구조가 불명확함
* sql-map-config 기반 참조 구조가 남아 있음

---

## 5. Spring XML 규칙 설계

### 핵심 규칙

#### FactoryBean 전환

```xml
org.springframework.orm.ibatis.SqlMapClientFactoryBean
```

→

```xml
org.mybatis.spring.SqlSessionFactoryBean
```

#### property 구조 조정

주요 대상:

* `configLocation`
* `mapperLocations`
* `sqlMapClient`
* DAO bean의 연계 property

### Spring XML 경고 규칙

아래 경우 경고를 남긴다.

* 기존 bean id 참조가 광범위하게 남아 있음
* `mapperLocations` 계산이 불명확함
* `lobHandler` 같은 특수 property가 남아 있음
* `SqlMapClient` 참조 bean이 다른 파일에 흩어져 있음

## 6. 후처리와 보고서 설계

### postcheck가 탐지해야 할 항목

* `EgovAbstractDAO`
* `SqlMapClientFactoryBean`
* `SqlMapClientTemplate`
* `#...#`
* `<dynamic>`
* `<isNotEmpty>`
* `<iterate>`

### 보고서가 보여줘야 할 항목

* 전체 finding 수
* 변환 파일 수
* 변경 파일 수
* 수동 검토 파일 수
* 경고 유형별 개수
* 대표 수동 검토 파일 목록

### 수동 검토 분류 기준

* 구조는 바뀌었지만 의미 검증이 필요한 경우
* 자동 변환이 일부만 된 경우
* 외부 설정/업무 SQL/커스텀 래퍼가 개입된 경우

---

## 7. 향후 확장 포인트

향후 규칙 확장은 아래 방향으로 진행한다.

* DAO API 패턴별 세분화
* `sqlMapConfig` 확장 규칙 고도화
* 동적 SQL 안전 변환 범위 확대
* Spring XML 다중 datasource 패턴 지원
* LLM 보정 대상과 Python 경고 체계 연결

---

## 8. 결론

Phase2 규칙 설계의 핵심은 "많이 바꾸는 것"보다 "안전하게 바꾸고, 애매한 것은 정확히 경고하는 것"이다.

따라서 규칙은 아래 세 층으로 운영하는 것이 적절하다.

* 자동 변환 규칙
* 조건부 변환 + 경고 규칙
* 수동 검토 / Phase3 연계 규칙
