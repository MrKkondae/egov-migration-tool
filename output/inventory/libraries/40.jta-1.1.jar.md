## 2. 라이브러리 개요

### 주요 기능

* Java 표준 트랜잭션 API 제공
* `UserTransaction`, `TransactionManager`, `Transaction` 등 트랜잭션 인터페이스 제공
* WAS 또는 트랜잭션 매니저와 연계한 전역 트랜잭션 처리 지원
* XA 기반 다중 리소스 트랜잭션 연계 지원

### 사용 목적

JTA(Java Transaction API)는 Java 애플리케이션에서 트랜잭션을 표준 방식으로 제어하기 위한 API이다.

일반적인 단일 DB 트랜잭션은 Spring의 `DataSourceTransactionManager`로 처리할 수 있으나, 다음과 같은 경우에는 JTA 검토가 필요하다.

* 2개 이상의 DB를 하나의 트랜잭션으로 묶는 경우
* DB와 JMS 등 복수 리소스를 하나의 트랜잭션으로 처리하는 경우
* JEUS, WebLogic 등 상용 WAS의 전역 트랜잭션 기능을 사용하는 경우
* XA DataSource를 사용하는 경우

따라서 본 라이브러리는 단순 버전업 대상이라기보다, 실제 전역 트랜잭션 사용 여부와 WAS 제공 API 여부를 기준으로 유지/제거/대체를 판단해야 한다.

### 대표 사용 예시

```java
import javax.transaction.UserTransaction;

public class SampleTransactionService {

    private UserTransaction userTransaction;

    public void process() throws Exception {
        userTransaction.begin();
        try {
            // 업무 처리
            userTransaction.commit();
        } catch (Exception e) {
            userTransaction.rollback();
            throw e;
        }
    }
}
```

Spring XML 설정에서는 다음과 같은 형태로 사용될 수 있다.

```xml
<bean id="transactionManager"
      class="org.springframework.transaction.jta.JtaTransactionManager" />
```

단일 DB 트랜잭션이면 다음과 같은 구성이 더 일반적이다.

```xml
<bean id="transactionManager"
      class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
    <property name="dataSource" ref="dataSource" />
</bean>
```

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 검토

### 전환 사유

`jta-1.1.jar`는 업무 기능 라이브러리가 아니라 트랜잭션 API 라이브러리이다.

전자정부프레임워크 4.3은 Spring Framework 5.3.37 및 JDK 1.8+ 기반으로 동작한다. Spring 기반 단일 DB 트랜잭션은 보통 `DataSourceTransactionManager`로 처리할 수 있으므로, JTA가 실제로 필요한지 먼저 확인해야 한다.

JEUS, WebLogic 등 상용 WAS는 JTA API와 트랜잭션 매니저를 자체 제공할 수 있다. 이 경우 애플리케이션 `WEB-INF/lib`에 `jta-1.1.jar`를 중복 배치하면 클래스 충돌 또는 버전 충돌 가능성이 있으므로 WAS 제공 라이브러리 사용 여부를 확인해야 한다.

### 영향 범위

| 구분      | 영향 |
| ------- | ---- |
| 소스      | `javax.transaction.*`, `UserTransaction` 직접 사용 시 영향 있음 |
| 설정파일    | `JtaTransactionManager`, JNDI, XA 설정 사용 시 영향 있음 |
| pom.xml | `javax.transaction:jta:1.1` 또는 `javax.transaction-api` 의존성 정리 필요 |
| 운영환경    | WAS 제공 JTA, XA DataSource, JNDI, DB 드라이버 설정 확인 필요 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* 단일 DB 트랜잭션인지, XA/분산 트랜잭션인지 구분
* `JtaTransactionManager` 사용 여부 확인
* `DataSourceTransactionManager`로 대체 가능한지 확인
* WAS가 JTA API를 제공하는지 확인
* 애플리케이션 lib와 WAS 공통 lib의 JTA 중복 여부 확인
* XA DataSource 사용 여부 확인
* DB, JMS, EAI 등 복수 리소스 트랜잭션 여부 확인
* 운영 WAS별 트랜잭션 timeout, rollback, recovery 설정 확인

### 자동 변환 가능 여부

| 구분          | 가능 여부 |
| ----------- | -------- |
| OpenRewrite | 제한적 가능 |
| Python Rule | 가능 |
| LLM 보정      | 필요 |

### 자동화 가능 범위

* `javax.transaction` import 검색
* `JtaTransactionManager` 설정 검색
* `jta-1.1.jar` 중복 배치 검색
* `pom.xml` 내 JTA 의존성 검색
* 단일 DB 트랜잭션 후보와 XA 트랜잭션 후보 목록화

### 자동화 어려운 범위

* 업무상 분산 트랜잭션 필요 여부 판단
* WAS 트랜잭션 매니저 설정 검증
* XA rollback/recovery 정상 동작 확인
* 장애 상황에서 트랜잭션 정합성 검증

### 예상 난이도

* 단일 DB 트랜잭션만 사용하는 경우: 낮음
* WAS 제공 JTA를 사용하는 경우: 중간
* XA/분산 트랜잭션을 사용하는 경우: 높음

## 6. 버전 업그레이드 검토

### AS-IS 버전

```text
javax.transaction:jta:1.1
또는
jta-1.1.jar
```

### TO-BE 버전

```text
WAS 제공 JTA API 사용
또는
javax.transaction:javax.transaction-api:1.3 계열 검토
또는
javax.transaction:transaction-api:1.1 계열 검토
```

※ 실제 적용 버전은 eGovFrame 4.3, JDK 1.8, WAS, DB 드라이버, 운영 표준 라이브러리 정책에 맞춰 확정해야 한다.

### 주요 변경사항

* `jta-1.1.jar`를 애플리케이션에 직접 포함할지, WAS 제공 API로 처리할지 결정 필요
* 단일 DB 트랜잭션은 `DataSourceTransactionManager`로 단순화 가능
* 전역 트랜잭션은 `JtaTransactionManager` 유지 및 WAS/JNDI/XA 설정 검증 필요
* `javax.transaction.*` 패키지는 eGovFrame 4.3의 `javax.*` 기반과 호환
* Tomcat 10 / Jakarta EE 전환이 아닌 경우 `jakarta.transaction.*`로 일괄 변경하지 않음

### 소스 수정 여부

* 단일 DB이고 JTA 미사용: 없음
* 설정에서만 JTA 사용: 일부 설정 수정 필요
* `UserTransaction` 직접 사용: 일부 소스 수정 필요
* XA/분산 트랜잭션 사용: 설정 및 운영 검증 필요

## 7. 위험요소

### 보안

* JTA API 자체는 인터페이스 성격이므로 직접적인 보안 취약점 영향은 제한적이다.
* 다만 트랜잭션 처리 오류는 데이터 정합성 문제로 이어질 수 있다.
* WAS 공통 lib와 애플리케이션 lib 중복으로 예기치 않은 클래스 로딩 문제가 발생할 수 있다.

### 운영

* `jta-1.1.jar`를 임의 제거하면 `NoClassDefFoundError: javax/transaction/...` 오류가 발생할 수 있다.
* WAS 제공 JTA와 애플리케이션 포함 JTA가 중복되면 클래스 충돌 가능성이 있다.
* XA 트랜잭션 사용 시 timeout, recovery, 2-phase commit, rollback 정책을 운영 환경에서 반드시 확인해야 한다.
* 단일 DB 트랜잭션인데 JTA를 계속 사용하면 설정이 불필요하게 복잡해질 수 있다.

### 호환성

| 항목            | 영향 |
| --------------- | ---- |
| JDK 1.8         | `javax.transaction` 계열 사용 가능 |
| Spring 5.3      | `DataSourceTransactionManager`, `JtaTransactionManager` 모두 사용 가능 |
| eGovFrame 4.3   | `javax.*` 기반 유지 판단 필요 |
| WAS             | JEUS/WebLogic 등 WAS 제공 JTA와 중복 확인 필요 |
| DB Driver       | XA 사용 시 XA 지원 드라이버 및 DataSource 설정 확인 필요 |
| Jakarta EE      | 별도 전환 대상이 아닌 경우 `jakarta.transaction` 전환 금지 |

## 9. 최종 판정

### 판정

```text
전환대상여부 : 검토

조치방안 :
- jta-1.1.jar 직접 사용 여부 확인
- 단일 DB 트랜잭션이면 Spring DataSourceTransactionManager로 처리 가능 여부 확인
- XA/분산 트랜잭션이면 WAS 제공 JTA 또는 javax.transaction-api 계열로 정리
- WAS 공통 lib와 애플리케이션 lib 중복 배치 제거
- 트랜잭션 commit/rollback 회귀 테스트 수행

자동변환 :
- 부분 가능

소스수정 :
- 사용 범위에 따라 없음 ~ 일부 수정

위험도 :
- 중간
```

### 비고

* 본 항목은 단순 버전업보다 사용 필요성 판단이 우선이다.
* 단일 DB 기반 업무라면 JTA를 제거하고 Spring `DataSourceTransactionManager`로 단순화하는 방안을 우선 검토한다.
* 다중 DB, JMS, EAI 등 복수 리소스를 하나의 트랜잭션으로 묶는 경우에는 JTA 유지가 필요할 수 있다.
* JEUS/WebLogic 등 상용 WAS 환경에서는 WAS가 제공하는 JTA API와 애플리케이션에 포함된 `jta-1.1.jar`의 중복 여부를 반드시 확인한다.
* eGovFrame 4.3이 `javax.*` 기반이므로 Tomcat 10/Jakarta EE 전환이 아닌 이상 `jakarta.transaction.*` 전환은 별도 과제로 분리한다.
