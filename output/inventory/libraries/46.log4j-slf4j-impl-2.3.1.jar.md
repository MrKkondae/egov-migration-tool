## 2. 라이브러리 개요

### 주요 기능

* SLF4J API 호출을 Log4j2 API 및 Log4j2 Core로 전달
* 애플리케이션 또는 외부 라이브러리의 `org.slf4j.Logger` 기반 로그를 Log4j2 구현체로 출력
* Log4j2를 최종 로깅 구현체로 사용할 때 SLF4J 연계 기능 제공
* Logback, slf4j-simple 등 다른 SLF4J Binding과의 중복 방지 필요

### 사용 목적

`log4j-slf4j-impl`은 SLF4J API를 사용하는 코드 또는 외부 라이브러리의 로그 호출을 Log4j2로 전달하기 위한 Binding 라이브러리이다.

일반적으로 다음 구성이 함께 사용된다.

```text
slf4j-api
  ↓
log4j-slf4j-impl
  ↓
log4j-api
  ↓
log4j-core
```

AS-IS의 `log4j-slf4j-impl-2.3.1.jar`는 Log4j2 2.3.x 계열에 속하며, Log4j2 보안 취약점 이슈와 버전 노후화 관점에서 유지하지 않는 것이 적절하다.

본 항목은 43번 `log4j-api-2.3.2.jar`, 44번 `log4j-core-2.3.2.jar`와 반드시 세트로 검토해야 한다.

### 대표 사용 예시

업무 소스에서는 보통 SLF4J API를 직접 사용하고, `log4j-slf4j-impl`은 런타임 Binding으로 동작한다.

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class SampleService {
    private static final Logger LOGGER = LoggerFactory.getLogger(SampleService.class);

    public void process() {
        LOGGER.info("업무 처리 시작");
    }
}
```

Maven 의존성 예시는 다음과 같다.

```xml
<dependency>
    <groupId>org.apache.logging.log4j</groupId>
    <artifactId>log4j-slf4j-impl</artifactId>
    <version>2.23.1</version>
</dependency>
```

## 3. AS-IS 사용 현황

### 프로젝트 내 사용 여부

* 확인 필요
* SLF4J API 사용 시 런타임 Binding으로 사용 가능성 높음
* `log4j-api`, `log4j-core`와 함께 구성되었을 가능성 높음

### 확인 방법

소스 내 SLF4J API 사용 여부 확인

```bash
grep -R "org.slf4j.Logger" src
grep -R "LoggerFactory" src
```

Log4j2 설정파일 확인

```bash
find . -name "log4j2*.xml" -o -name "log4j2*.properties" -o -name "log4j2*.json" -o -name "log4j2*.yaml"
```

중복 Binding 확인

```bash
find . -name "*.jar" | grep -E "slf4j|log4j|logback"
```

Maven Dependency Tree 확인

```bash
mvn dependency:tree -Dincludes=org.slf4j,org.apache.logging.log4j,ch.qos.logback
```

동시 사용 금지 또는 충돌 가능성이 있는 라이브러리 확인

```bash
find . -name "log4j-to-slf4j*.jar"
find . -name "slf4j-log4j12*.jar"
find . -name "slf4j-reload4j*.jar"
find . -name "logback-classic*.jar"
find . -name "slf4j-simple*.jar"
```

### 예상 사용 위치

* 공통 로깅 설정
* Spring Framework 내부 및 외부 라이브러리 로그 출력
* Service / Controller / DAO / Batch 전 영역
* WAS 기동 로그 및 애플리케이션 업무 로그
* 배치 로그, 오류 로그, SQL 로그 연계 영역

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상
* 버전업 대상
* 보안 조치 대상

### 전환 사유

`log4j-slf4j-impl-2.3.1.jar`는 Log4j2 2.3.x 계열의 오래된 Binding 라이브러리이다.

Log4j2는 API, Core, Bridge/Binding 모듈이 서로 맞물려 동작하므로 `log4j-slf4j-impl`만 단독으로 버전업해서는 안 된다. `log4j-api`, `log4j-core`, `log4j-slf4j-impl` 버전을 동일한 Log4j2 계열로 통일해야 한다.

eGovFrame 4.3 Lite 라이브러리 목록에는 `log4j-api 2.23.1`, `log4j-core 2.23.1`, `log4j-slf4j-impl 2.23.1`이 포함되어 있으므로, 사업 표준 라이브러리 기준이 확정되지 않은 경우 2.23.1 계열을 우선 후보로 검토할 수 있다.

Matrix에는 TO-BE 기준이 `log4j-slf4j-impl-2.17.1+ 또는 eGov 4.3 검증 버전`으로 정리되어 있으므로, 최소 기준은 2.17.1 이상이며 최종 기준은 eGovFrame 4.3 검증 버전 또는 프로젝트 표준 버전으로 확정한다.

### 영향 범위

| 구분      | 영향 |
| ------- | ---- |
| 소스      | 대부분 없음. 단, SLF4J 2.x 또는 Log4j2 API 직접 사용 방식 변경 시 일부 수정 가능 |
| 설정파일    | Log4j2 설정파일, SLF4J Binding, WAS 공통 라이브러리 구성 확인 필요 |
| pom.xml | `log4j-slf4j-impl`, `log4j-api`, `log4j-core` 버전 통일 필요 |
| 운영환경    | 로그 출력, 로그 롤링, 배치 로그, 장애 추적 로그 영향 가능 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* `log4j-slf4j-impl-2.3.1.jar` 제거
* `log4j-slf4j-impl`을 2.17.1 이상 또는 eGovFrame 4.3 검증 버전으로 버전업
* `log4j-api`, `log4j-core`, `log4j-slf4j-impl` 버전 계열 통일
* `log4j-to-slf4j`와 동시 사용 금지
* `slf4j-log4j12`, `slf4j-reload4j`, `logback-classic`, `slf4j-simple` 등 다른 SLF4J Binding 중복 여부 확인
* WAS 공통 lib와 애플리케이션 `WEB-INF/lib` 중복 배치 확인
* `log4j2.xml`, `log4j2.properties` 등 설정파일 호환성 확인
* 로그 출력, 로그 레벨, 로그 파일 경로, Rolling 정책 회귀 테스트 수행

### 자동 변환 가능 여부

| 구분          | 가능 여부 |
| ----------- | -------- |
| OpenRewrite | 부분 가능 |
| Python Rule | 가능 |
| LLM 보정      | 필요 |

### 예상 난이도

* 중간

단순 의존성 버전업 자체는 난이도가 높지 않다. 그러나 로깅 라이브러리는 전 구간에 영향을 주며 중복 Binding, 순환 Bridge, WAS 공통 lib 중복 배치, 보안점검 결과와 연결되므로 운영 검증 난이도는 중간 이상으로 보는 것이 적절하다.

## 6. 버전 업그레이드 검토

### AS-IS 버전

```text
org.apache.logging.log4j:log4j-slf4j-impl:2.3.1
```

### TO-BE 버전

```text
org.apache.logging.log4j:log4j-slf4j-impl:2.17.1 이상
또는
org.apache.logging.log4j:log4j-slf4j-impl:2.23.1 (eGovFrame 4.3 Lite 목록 기준)
또는
프로젝트 보안검증 및 표준 라이브러리 목록에서 확정한 Log4j2 검증 버전
```

### 주요 변경사항

* Log4j2 2.3.x 계열에서 보안 조치 버전으로 상향
* Log4j2 API/Core/Binding 버전 정합성 확보
* SLF4J 1.7.x 사용 시 `log4j-slf4j-impl` 사용
* SLF4J 2.x 사용 시 `log4j-slf4j2-impl` 검토 필요
* `log4j-to-slf4j`와 `log4j-slf4j-impl` 동시 사용 금지
* Log4j2 설정파일 및 Appender 동작 검증 필요

### 소스 수정 여부

* 대부분 없음
* SLF4J 1.7.x 유지 시 소스 수정 영향 낮음
* SLF4J 2.x로 함께 전환하거나 Log4j2 API 직접 사용 구조를 표준화하는 경우 일부 수정 가능

## 7. 위험요소

### 보안

* Log4j2 2.3.x 계열은 보안점검에서 취약 라이브러리로 식별될 가능성이 높음
* Log4Shell 계열 취약점은 주로 `log4j-core`가 핵심 영향 모듈이지만, 운영 배포 단위에서는 `log4j-api`, `log4j-core`, `log4j-slf4j-impl`을 세트로 관리해야 함
* SBOM, 보안점검 결과, 폐쇄망 반입 라이브러리 목록에 동일 버전 기준 반영 필요

### 운영

* 잘못된 Binding 조합은 로그 미출력, 중복 출력, 기동 오류, 무한 순환 오류를 유발할 수 있음
* `log4j-slf4j-impl`과 `log4j-to-slf4j`를 동시에 배치하면 순환 의존성 문제가 발생할 수 있음
* WAS 공통 lib와 애플리케이션 lib에 다른 버전의 Log4j2가 동시에 존재하면 예측하기 어려운 로깅 오류 가능
* 로그 파일 경로, 권한, Rolling 정책, 일자별 로그 분리 정책 확인 필요

### 호환성

| 항목            | 영향 |
| ------------- | ---- |
| JDK 8         | Log4j2 2.17.1 이상 및 2.23.1 계열 적용 가능 범위. 프로젝트 JDK 기준으로 검증 필요 |
| Spring 5.3    | 직접 영향 낮음. Spring 내부 로그와 SLF4J/JCL Bridge 구성 확인 필요 |
| eGovFrame 4.3 | eGovFrame Lite 4.3 목록 기준 Log4j2 2.23.1 계열 확인됨 |
| SLF4J         | SLF4J 1.7.x는 `log4j-slf4j-impl`, SLF4J 2.x는 `log4j-slf4j2-impl` 검토 필요 |
| WAS           | 공통 lib 중복 배치 및 클래스 로딩 우선순위 확인 필요 |

## 9. 최종 판정

### 판정

```text
전환대상여부 : 대상 / 버전업 / 보안

조치방안 :
- log4j-slf4j-impl-2.3.1.jar를 2.17.1 이상 또는 eGovFrame 4.3 검증 버전으로 버전업
- log4j-api, log4j-core, log4j-slf4j-impl 버전을 동일 계열로 통일
- eGovFrame 4.3 Lite 기준 후보는 2.23.1
- log4j-to-slf4j와 동시 사용 금지
- 다른 SLF4J Binding 중복 여부 확인
- 로그 출력 및 보안점검 회귀 테스트 수행

자동변환 :
- 부분 가능
- 의존성 변경, JAR 중복 검색, 설정파일 검색은 자동화 가능
- 최종 로그 구현체 선정과 운영 로그 검증은 수동 확인 필요

소스수정 :
- 대부분 없음
- SLF4J 2.x 전환 또는 로그 표준화 시 일부 수정 가능

위험도 :
- 높음
```

### 비고

* 본 항목은 43번 `log4j-api-2.3.2.jar`, 44번 `log4j-core-2.3.2.jar`와 세트로 검토해야 한다.
* Log4j2 관련 라이브러리는 보안점검에서 중요한 항목이므로 마이그레이션 작업과 별개로 선제 패치 대상이다.
* 최종 버전은 `2.17.1 이상`을 최소 기준으로 하되, eGovFrame 4.3 검증 라이브러리 목록과 프로젝트 표준 라이브러리 목록에 맞춰 확정한다.
* SLF4J 1.7.x 기반이면 `log4j-slf4j-impl`, SLF4J 2.x 기반이면 `log4j-slf4j2-impl` 적용 여부를 별도 검토한다.
* 폐쇄망 반입 시 `log4j-api`, `log4j-core`, `log4j-slf4j-impl` 버전 불일치가 발생하지 않도록 라이브러리 목록을 함께 관리해야 한다.
