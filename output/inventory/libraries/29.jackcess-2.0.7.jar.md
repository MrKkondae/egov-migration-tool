## 2. 라이브러리 개요

### 주요 기능

* Microsoft Access 데이터베이스 파일(`.mdb`, `.accdb`) 읽기 및 쓰기
* Access 파일의 테이블, 컬럼, 행(Row), 인덱스, 관계 등 구조 정보 접근
* 별도 MS Access 또는 ODBC/JDBC 드라이버 없이 Java API로 Access 파일 직접 처리
* Access 파일 기반 자료 반입, 변환, 배치성 데이터 처리 지원

### 사용 목적

Jackcess는 Java 애플리케이션에서 Microsoft Access 파일을 직접 읽고 쓰기 위한 순수 Java 라이브러리이다.
일반적인 업무시스템의 주 데이터베이스 연계 라이브러리라기보다는, 외부 기관 또는 사용자로부터 전달받은 `.mdb`, `.accdb` 파일을 읽어 업무 DB로 반입하거나, Access 파일 형식으로 데이터를 생성·내보내는 용도로 사용된다.

전자정부프레임워크 4.3 전환 관점에서는 핵심 프레임워크 라이브러리는 아니지만, 실제 소스에서 Access 파일 처리 기능이 존재할 경우 업무 데이터 반입·배치·파일 처리 기능에 직접 영향을 줄 수 있다.
따라서 무조건 유지가 아니라, 실제 사용 여부를 먼저 확인한 뒤 다음 중 하나로 조치하는 것이 타당하다.

* 실제 사용 중이면 JDK 1.8 이상 호환 Jackcess 상위 버전으로 버전업
* 사용하지 않으면 의존성 제거
* 암호화된 Access 파일을 처리하는 경우 `jackcess-encrypt` 사용 여부까지 별도 확인

### 대표 사용 예시

```java
import java.io.File;

import com.healthmarketscience.jackcess.Database;
import com.healthmarketscience.jackcess.DatabaseBuilder;
import com.healthmarketscience.jackcess.Row;
import com.healthmarketscience.jackcess.Table;

public class AccessFileReader {
    public void readAccessFile() throws Exception {
        try (Database db = DatabaseBuilder.open(new File("sample.mdb"))) {
            Table table = db.getTable("SAMPLE_TABLE");

            for (Row row : table) {
                System.out.println(row.get("ID"));
                System.out.println(row.get("NAME"));
            }
        }
    }
}
```

### 확인 방법

#### 1) Java import 및 API 사용처 확인

```bash
grep -R "com.healthmarketscience.jackcess" ./src
grep -R "DatabaseBuilder" ./src
grep -R "Database.open" ./src
grep -R "getTable(" ./src
grep -R "jackcess" ./src
```

#### 2) Access 파일 확장자 사용처 확인

```bash
grep -R "\.mdb" ./src ./config ./resources
grep -R "\.accdb" ./src ./config ./resources
grep -R "access" ./src ./config ./resources
```

#### 3) Maven/Gradle 의존성 확인

```bash
mvn dependency:tree -Dincludes=com.healthmarketscience.jackcess
mvn dependency:tree | grep -i jackcess
```

#### 4) 암호화 Access 파일 처리 여부 확인

```bash
grep -R "jackcess-encrypt" ./
grep -R "CryptCodecProvider" ./src
grep -R "CodecProvider" ./src
```

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상
* 단, 실제 사용 여부 확인 후 `버전업` 또는 `제거`로 최종 판정

### 전환 사유

`jackcess-2.0.7.jar`는 2014년에 배포된 구버전 라이브러리이며, 전자정부프레임워크 4.3 전환 시 JDK 1.8 이상 환경과 오픈소스 라이선스 기준을 함께 검토해야 한다.

특히 Jackcess 2.0.7은 Maven Repository 기준 LGPL 2.1 라이선스로 표시된다. 반면 Jackcess 프로젝트는 2.1.0부터 Apache License 2.0으로 재라이선스되었으므로, 공공기관 사업에서는 기존 2.0.7을 유지하기보다 Apache License 2.0 계열의 상위 버전 적용을 우선 검토하는 것이 안전하다.

다만 Jackcess는 업무 프레임워크 핵심 라이브러리가 아니며, 실제로 `.mdb` 또는 `.accdb` 파일을 처리하는 업무 기능이 없으면 제거 가능하다.
따라서 본 항목은 단순 버전업 대상이 아니라 사용처 확인이 선행되어야 하는 `버전업 / 검토` 항목으로 분류한다.

### 영향 범위

| 구분 | 영향 |
| --- | --- |
| 소스 | Access 파일 처리 API 사용처가 있으면 일부 수정 가능 |
| 설정파일 | 파일 업로드 경로, 임시파일 경로, 배치 입력 경로 설정 확인 필요 |
| pom.xml | `jackcess-2.0.7.jar` 직접 참조 제거 후 Maven 의존성으로 관리 권장 |
| 운영환경 | Access 파일 입출력 경로, 파일 권한, 문자셋, 대용량 파일 처리, 암호화 파일 여부 확인 필요 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* 프로젝트 내 `.mdb`, `.accdb` 파일 처리 기능 존재 여부 확인
* `com.healthmarketscience.jackcess` 패키지 직접 사용 여부 확인
* `jackcess-encrypt` 등 암호화 Access 파일 처리 모듈 사용 여부 확인
* 기존 2.0.7의 LGPL 2.1 라이선스 반입 가능 여부 확인
* Apache License 2.0 계열 상위 버전으로 변경 시 API 및 결과값 차이 확인
* Access 파일 내 한글, 날짜, 숫자, 통화, 메모 필드, 첨부 필드 처리 회귀 테스트 수행
* 읽기 전용 기능인지, 쓰기/생성/수정 기능까지 사용하는지 확인
* 운영 서버에서 파일 경로, 파일 잠금, 동시 처리, 대용량 처리에 문제가 없는지 확인

### 자동 변환 가능 여부

| 구분 | 가능 여부 |
| --- | --- |
| OpenRewrite | 제한적 가능. Maven 의존성 버전 변경 정도는 가능하나 업무 로직 검증은 불가 |
| Python Rule | 가능. import, 파일 확장자, 의존성, API 사용처 목록화 가능 |
| LLM 보정 | 필요. API 변경, 예외 처리, 날짜/문자셋 처리, 암호화 파일 처리 여부는 수동 검토 필요 |

### 예상 난이도

* 중간

단순히 파일을 읽는 기능만 있으면 난이도는 낮음~중간 수준이다.
하지만 Access 파일 생성, 수정, 암호화 파일 처리, 대량 데이터 반입, 업무 DB 적재 로직과 연계되어 있으면 난이도는 중간 이상으로 상승한다.

### TO-BE 버전 선정 기준

| 후보 | 검토 의견 |
| --- | --- |

| 2.2.x | 2.x API 유지 가능성이 높아 보수적 전환 시 검토 가능. 단, 최신성은 낮음 |
| 2.1.x | 2.1.0부터 Apache License 2.0으로 재라이선스된 계열. 라이선스 리스크 완화 목적의 최소 버전업 후보 |
| 제거 | 실제 `.mdb`, `.accdb` 사용처가 없으면 가장 안전한 조치 |

### 주요 변경사항

* 2.0.7은 LGPL 2.1로 표시되나, 2.1.0부터 Apache License 2.0으로 재라이선스됨
* 3.0.0부터 Java 8+ 요구 및 Java 8 날짜/경로 타입 지원이 추가됨
* 4.0.0부터 Java 9+ 모듈 경로 사용을 위한 `Automatic-Module-Name`이 추가됨
* 4.0.0은 3.x 계열과 바이너리 호환성이 안내되어 있으나, 2.0.7에서 직접 4.x로 올릴 경우 2.x → 3.x 전환 영향은 별도 검증 필요
* 날짜/시간, 문자셋, 정렬, 인덱스, Access 버전별 파일 포맷 처리 결과가 달라질 수 있으므로 샘플 파일 기반 회귀 테스트 필요

### 소스 수정 여부

* 사용처 없음: 소스 수정 없음, 의존성 제거
* 단순 읽기 사용: 대부분 유지 가능하나 일부 예외 처리 및 API 호환성 확인 필요
* 쓰기/생성/암호화 파일 처리: 일부 수정 필요 가능성 높음
* UCanAccess와 함께 사용 중인 경우: UCanAccess 버전과 Jackcess 버전 호환성 별도 검토 필요

## 7. 위험요소

### 보안

* Jackcess 2.0.7 자체의 직접 취약점은 별도 SCA 도구로 확인 필요
* Maven Repository에는 2.0.7의 전이 의존성 취약점이 표시되므로, 운영 반입 전 의존성 전체에 대한 보안 점검 필요
* 파일 업로드를 통해 Access 파일을 수신하는 구조라면 파일 확장자, MIME 타입, 저장 경로, 파일 크기 제한, 악성 파일 검사 필요
* Access 파일 내용을 업무 DB에 적재하는 경우 SQL Injection보다 데이터 검증, 인코딩, 필드 길이 초과, 비정상 데이터 처리 위험이 중요함

### 운영

* `.mdb`, `.accdb` 파일이 실제 운영 연계에 사용되는지 확인 필요
* 대용량 Access 파일 처리 시 메모리 사용량과 처리 시간이 증가할 수 있음
* 파일 잠금, 동시 처리, 임시파일 삭제, 배치 재처리 기준 확인 필요
* 한글 문자셋, 날짜/시간, 통화, 숫자 정밀도, 메모 필드 처리 결과 검증 필요
* 암호화된 Access 파일은 기본 Jackcess만으로 처리되지 않을 수 있어 `jackcess-encrypt` 또는 대체 처리 방식 확인 필요

### 호환성

| 항목 | 영향 |
| --- | --- |
| JDK 1.8 | 3.0.0 이상 Jackcess는 Java 8+ 요구. eGovFrame 4.3 실행환경과 호환 가능하나 실제 WAS/JDK 조합에서 테스트 필요 |
| JDK 17 | 4.x 계열은 Java 9+ 모듈 경로 호환성이 안내되어 JDK 17 환경에서도 검증 가능 |
| Spring 5.3 | 직접 영향 낮음. Spring Batch, 파일 업로드, 스케줄러와 연계된 경우 업무 테스트 필요 |
| eGovFrame 4.3 | 프레임워크 핵심 의존성은 아니지만, 파일 반입/배치/외부자료 처리 업무에 영향 가능 |
| WAS | 파일 경로, 권한, 임시 디렉터리, 업로드 제한 설정 확인 필요 |
