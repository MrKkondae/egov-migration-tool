## 2. 라이브러리 개요

### 주요 기능

* Java 애플리케이션 로깅 API 제공
* Logger, LogManager, Marker, ThreadContext 등 로그 호출 인터페이스 제공
* Log4j Core, Log4j SLF4J Binding 등 로그 구현체와 연계
* 애플리케이션 코드와 로그 구현체를 분리하는 API 계층 제공

### 사용 목적

`log4j-api`는 Apache Log4j 2의 API 모듈이다. 애플리케이션 소스는 `org.apache.logging.log4j.Logger`, `LogManager` 등을 통해 로그를 호출하고, 실제 출력 처리는 `log4j-core` 또는 SLF4J 연계 구현체가 담당한다.

`log4j-api-2.3.2.jar` 자체는 API 계층이지만, 운영 환경에서는 일반적으로 `log4j-core`, `log4j-slf4j-impl`, `log4j-over-slf4j` 등과 함께 구성된다. 따라서 단일 JAR만 판단하지 말고 Log4j2 전체 구성의 버전 정합성과 보안 취약점 조치 여부를 함께 확인해야 한다.

### 대표 사용 예시

```java
import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;

public class SampleService {
    private static final Logger LOGGER = LogManager.getLogger(SampleService.class);

    public void process(String userId) {
        LOGGER.info("process start. userId={}", userId);
    }
}
```

## 3. AS-IS 사용 현황

### 프로젝트 내 사용 여부

* 확인 필요
* Log4j2 API를 직접 import하여 사용할 가능성 있음
* Log4j2 Core 또는 SLF4J Binding의 의존성으로 포함되었을 가능성 있음

### 확인 방법

#### 소스 사용처 확인

```bash
grep -R "org.apache.logging.log4j" ./src
grep -R "LogManager" ./src
grep -R "ThreadContext" ./src
grep -R "MarkerManager" ./src
```

#### 설정파일 확인

```bash
find . -iname "log4j2*.xml" -o -iname "log4j2*.properties" -o -iname "log4j2*.yml" -o -iname "log4j2*.yaml"
grep -R "JndiLookup\|JDBC\|JMS\|Socket\|SMTP\|Syslog" ./src ./resources 2>/dev/null
```

#### 라이브러리 중복 확인

```bash
find . -iname "log4j*.jar" -o -iname "slf4j*.jar" -o -iname "logback*.jar"
jar tf WEB-INF/lib/log4j-api-2.3.2.jar | head
mvn dependency:tree -Dincludes=org.apache.logging.log4j
```

### 예상 사용 위치

* 공통 로깅 유틸리티
* Service / DAO / Controller

* AOP 로깅
* 예외 처리 공통 모듈

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상
* 버전업 / 보안 조치 대상

### 전환 사유

`log4j-api-2.3.2.jar`는 구버전 Log4j2 API이다. Log4j2 2.3.x 계열은 과거 보안 취약점 조치와 관련하여 버전 확인 및 패치가 필요한 영역이다.

특히 Log4Shell 계열 취약점은 주로 `log4j-core`에 영향을 주는 것으로 알려져 있으나, 실제 운영 구성에서는 `log4j-api`, `log4j-core`, `log4j-slf4j-impl`이 함께 배포되는 경우가 많다. 따라서 `log4j-api`만 단독 유지하지 말고 관련 Log4j2 모듈 버전을 동일 계열로 통일해야 한다.

Matrix 기준 TO-BE는 `log4j-api 2.17.1 이상 또는 eGovFrame 4.3 검증 버전`이며, eGovFrame Lite 4.3 목록에서는 `log4j-api 2.23.1`, `log4j-core 2.23.1`, `log4j-slf4j-impl 2.23.1`이 확인된다.

### 영향 범위

| 구분 | 영향 |
| --- | --- |
| 소스 | Log4j2 API 직접 사용 시 대부분 영향 낮음. 단, 제거 후 SLF4J API로 표준화할 경우 import 수정 필요 |
| 설정파일 | `log4j2.xml`, `log4j2.properties` 등 Log4j2 설정 검토 필요 |
| pom.xml | `log4j-api`, `log4j-core`, `log4j-slf4j-impl` 버전 통일 필요 |
| 운영환경 | WAS 공통 lib와 애플리케이션 lib 중복 배치, 로그 출력 경로, 파일 권한, 보안 패치 확인 필요 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* `log4j-api-2.3.2.jar`를 `2.17.1 이상` 또는 eGovFrame 4.3 검증 버전으로 버전업한다.
* `log4j-core`, `log4j-slf4j-impl`, `log4j-over-slf4j` 등 관련 모듈과 버전을 통일한다.
* `log4j-api`만 교체하지 말고 Log4j2 구성 전체를 함께 검토한다.
* `log4j-to-slf4j`와 `log4j-slf4j-impl`의 동시 사용 여부를 확인한다. 두 모듈이 함께 구성되면 로그 순환 위임 문제가 발생할 수 있다.
* `commons-logging`, `jcl-over-slf4j`, `log4j-over-slf4j`, `slf4j-api`, `logback` 등 로깅 브리지와 구현체 중복 여부를 확인한다.
* 보안 점검 시 `log4j-core` 존재 여부와 실제 버전을 함께 확인한다.
* 폐쇄망 반입 시 취약점 조치 완료 버전의 JAR와 SBOM 또는 라이브러리 목록을 산출물로 관리한다.

### 자동 변환 가능 여부

| 구분 | 가능 여부 |
| --- | --- |
| OpenRewrite | 부분 가능 |
| Python Rule | 가능 |
| LLM 보정 | 필요 |

### 예상 난이도

* 중간

단순 버전업만 수행하는 경우 난이도는 낮지만, 프로젝트 로깅 체계를 SLF4J/Logback 중심으로 재정리하거나 Log4j2 설정파일을 변경하는 경우 영향도가 커질 수 있다.

## 6. 버전 업그레이드 검토

### AS-IS 버전

```text
org.apache.logging.log4j:log4j-api:2.3.2
```

### TO-BE 버전

```text
1차 기준: org.apache.logging.log4j:log4j-api:2.17.1 이상
권장 검토: eGovFrame 4.3 검증 버전 또는 eGovFrame Lite 4.3 목록의 2.23.1 계열
```

### 주요 변경사항

* Log4j2 보안 취약점 조치 버전 반영
* `log4j-api`, `log4j-core`, `log4j-slf4j-impl` 버전 정합성 확보
* Log4j2 설정파일의 Appender, Layout, Lookup, JNDI 관련 설정 점검
* 운영 환경 로그 출력 방식 및 파일 롤링 정책 검증

### 소스 수정 여부

* 대부분 없음
* `org.apache.logging.log4j` API를 계속 사용할 경우 import 변경은 일반적으로 불필요
* SLF4J 표준화로 방향을 바꾸는 경우 `org.apache.logging.log4j` → `org.slf4j` import 수정 필요
* Log4j2 설정 API 또는 커스텀 Appender/Plugin을 직접 사용하는 경우 일부 수정 가능

## 7. 위험요소

### 보안

* Log4j2 2.3.x는 보안 취약점 이력으로 인해 선제 조치가 필요한 영역이다.
* 대표적으로 CVE-2021-44228, CVE-2021-45046, CVE-2021-44832 등 Log4j2 취약점이 존재한다.
* Apache 공식 보안 안내에 따르면 Log4Shell 계열 취약점의 영향 대상은 `log4j-core`이며, `log4j-api`만 사용하는 애플리케이션은 해당 취약점의 직접 영향 대상이 아니라고 안내되어 있다.
* 그러나 실제 운영환경에서는 `log4j-api`와 `log4j-core`가 함께 배포되는 경우가 많으므로, 보안 점검은 반드시 Log4j2 전체 모듈 기준으로 수행해야 한다.

### 운영

* 버전 불일치 시 로그 초기화 실패, 로그 미출력, 바인딩 충돌, ClassNotFoundException이 발생할 수 있다.
* WAS 공통 lib와 애플리케이션 lib에 서로 다른 Log4j2 버전이 중복 배치되면 예측하기 어려운 로그 동작이 발생할 수 있다.
* 파일 Appender 사용 시 로그 디렉터리 권한, 인코딩, 롤링 정책, 운영 로그 보관 정책을 함께 확인해야 한다.

### 호환성

| 항목 | 영향 |
| --- | --- |
| JDK 1.8 | `2.17.1` 이상 또는 eGovFrame 검증 버전 적용 가능 여부 확인 필요 |
| Spring 5.3 | 일반 로그 API 사용은 영향 낮음 |
| eGovFrame 4.3 | eGovFrame Lite 4.3 기준 Log4j2 2.23.1 계열 확인됨 |
| SLF4J | `slf4j-api`, `log4j-slf4j-impl`, `jcl-over-slf4j` 버전 정합성 확인 필요 |
| WAS | 공통 lib 중복 배치 및 클래스 로딩 우선순위 확인 필요 |

## 9. 최종 판정

### 판정

```text
전환대상여부 : 대상 / 버전업 / 보안

조치방안 :
- log4j-api-2.3.2.jar를 2.17.1 이상 또는 eGovFrame 4.3 검증 버전으로 버전업
- log4j-core, log4j-slf4j-impl과 동일 버전 계열로 통일
- WAS 공통 lib와 WEB-INF/lib 중복 배치 제거
- Log4j2 설정파일 및 로그 출력 회귀 테스트 수행
- 보안점검 시 log4j-api 단독이 아니라 Log4j2 전체 모듈 기준으로 확인

자동변환 :
- 부분 가능

소스수정 :
- 대부분 없음
- SLF4J 표준화 또는 커스텀 Log4j2 API 사용 시 일부 수정 필요

위험도 :
- 높음
```

### 비고

* 43번 `log4j-api-2.3.2.jar`, 44번 `log4j-core-2.3.2.jar`, 46번 `log4j-slf4j-impl-2.3.1.jar`는 개별 검토가 아니라 하나의 Log4j2 구성으로 묶어 버전 정합성을 확보해야 한다.
* Log4Shell 계열 취약점은 주로 `log4j-core` 영향이지만, 운영 배포 시 `log4j-api`와 `log4j-core`가 함께 존재하는 경우가 일반적이므로 `log4j-api`도 보안 버전업 대상으로 관리한다.
* eGovFrame Lite 4.3 목록에서는 Log4j2 2.23.1 계열이 확인되므로, 프로젝트 표준 라이브러리 목록과 폐쇄망 반입 기준에 맞춰 최종 버전을 확정해야 한다.
* 기존 로그 포맷, 로그 파일명, 로그 레벨, 파일 롤링 정책, 배치 로그 출력 위치는 업무 운영에 직접 영향이 있으므로 회귀 테스트가 필요하다.

## 참고 근거

* Library Migration Matrix: 43번 `log4j-api-2.3.2.jar`는 `버전업 / 보안` 대상으로 정리되어 있으며, `log4j-core`, `log4j-slf4j-impl`과 동일 버전 계열 통일이 필요하다고 되어 있다.
* Apache Log4j 공식 문서: Log4j2는 API와 구현체 Core로 분리되며, 애플리케이션은 API에 코딩하는 구조이다.
* Apache Logging Services 보안 공지: Log4Shell 계열 취약점은 `log4j-core`에 영향이 있으며, `log4j-api`만 사용하는 경우 직접 영향 대상이 아니라고 설명한다.
* eGovFrame Lite 4.3 가이드: `log4j-api 2.23.1`, `log4j-core 2.23.1`, `log4j-slf4j-impl 2.23.1` 항목이 확인된다.
