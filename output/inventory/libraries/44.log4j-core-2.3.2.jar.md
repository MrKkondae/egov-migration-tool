## 2. 라이브러리 개요

### 주요 기능

* Log4j2의 실제 로깅 구현체 제공
* Appender, Layout, Filter, LoggerContext 등 로그 처리 기능 제공
* 파일 로그, 콘솔 로그, 롤링 로그, 비동기 로그 등 출력 기능 제공
* `log4j2.xml`, `log4j2.properties` 기반 로그 설정 처리
* Log4j2 플러그인, Lookup, Appender 확장 기능 제공

### 사용 목적

`log4j-core`는 Apache Log4j 2에서 실제 로그 출력과 설정 해석을 담당하는 핵심 구현 모듈이다. 애플리케이션 소스는 일반적으로 `log4j-api` 또는 SLF4J API를 통해 로그를 호출하고, `log4j-core`가 로그 레벨, 출력 대상, 로그 포맷, 파일 롤링 정책 등을 처리한다.

`log4j-core-2.3.2.jar`는 과거 Java 6 호환 계열에서 사용되던 구버전 Log4j2 Core이다. 전자정부프레임워크 4.3 전환 환경에서는 JDK 1.8 이상을 기준으로 하므로, 구버전 2.3.x 계열을 유지하기보다 eGovFrame 4.3 검증 라이브러리 또는 보안 조치가 반영된 상위 버전으로 통일하는 것이 바람직하다.

### 대표 사용 예시

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Configuration status="WARN">
    <Appenders>
        <Console name="Console" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} %-5p %c - %m%n" />
        </Console>
    </Appenders>
    <Loggers>
        <Root level="info">
            <AppenderRef ref="Console" />
        </Root>
    </Loggers>
</Configuration>
```

실제 프로젝트에서는 `log4j2.xml`, `log4j2.properties` 등의 설정파일과 함께 사용되며, 업무 소스가 직접 `log4j-core` 클래스를 참조하는 경우는 많지 않다.

## 3. AS-IS 사용 현황

### 프로젝트 내 사용 여부

* 확인 필요
* Log4j2 로그 구현체로 사용 중일 가능성 높음
* `log4j-api-2.3.2.jar`, `log4j-slf4j-impl-2.3.1.jar`와 함께 구성되었을 가능성 높음

### 확인 방법

#### 라이브러리 존재 여부 확인

```bash
find . -iname "log4j-core*.jar"
find . -iname "log4j-api*.jar" -o -iname "log4j-slf4j-impl*.jar" -o -iname "log4j-over-slf4j*.jar"
find . -iname "slf4j*.jar" -o -iname "logback*.jar" -o -iname "commons-logging*.jar"
```

#### Maven 의존성 확인

```bash
mvn dependency:tree -Dincludes=org.apache.logging.log4j
mvn dependency:tree | grep -i "log4j\|slf4j\|logback\|commons-logging"
```

#### 설정파일 확인

```bash
find . -iname "log4j2*.xml" -o -iname "log4j2*.properties" -o -iname "log4j2*.yml" -o -iname "log4j2*.yaml"
grep -R "JndiLookup\|JDBC\|JMS\|Socket\|SMTP\|Syslog\|RollingFile\|Async" ./src ./resources 2>/dev/null
```

#### 보안 취약 클래스 확인

```bash
jar tf WEB-INF/lib/log4j-core-2.3.2.jar | grep "JndiLookup"
jar tf WEB-INF/lib/log4j-core-2.3.2.jar | grep "org/apache/logging/log4j/core"
```

### 예상 사용 위치

* 공통 로그 설정
* `log4j2.xml`, `log4j2.properties` 등 설정파일
* 업무 로그 / 오류 로그 / 배치 로그
* AOP 기반 공통 로깅
* SQL 로그 또는 연계 로그 출력 구성
* WAS 공통 lib 또는 애플리케이션 `WEB-INF/lib`

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상
* 버전업 / 보안 조치 대상

### 전환 사유

`log4j-core-2.3.2.jar`는 Log4j2의 실제 구현체이므로 보안 영향도가 `log4j-api`보다 높다. Log4Shell 계열 취약점은 `log4j-core`를 중심으로 논의되며, 운영환경에 `log4j-core`가 존재하면 버전, 설정, 취약 클래스, Appender 구성, 외부 입력 로그 출력 여부를 함께 확인해야 한다.

Matrix 기준 TO-BE는 `log4j-core 2.17.1 이상 또는 eGovFrame 4.3 검증 버전`이다. eGovFrame Lite 4.3 라이브러리 목록에서는 `log4j-api 2.23.1`, `log4j-core 2.23.1`, `log4j-slf4j-impl 2.23.1`이 확인되므로, 43번 `log4j-api`, 46번 `log4j-slf4j-impl`과 같은 버전 계열로 통일해야 한다.

단, Apache 보안 공지 기준으로 `2.3.2`는 일부 취약점에 대한 Java 6 계열 보안 패치 릴리스 성격도 있으므로, “무조건 Log4Shell 직접 취약 버전”이라고 단정하기보다 프로젝트 보안점검 도구, 실제 배포 JAR, 설정파일, 취약 클래스 포함 여부를 기준으로 확인해야 한다. 그럼에도 eGovFrame 4.3 전환 기준에서는 구버전 2.3.x 유지보다 검증된 상위 버전으로 표준화하는 것이 적절하다.

### 영향 범위

| 구분 | 영향 |
| --- | --- |
| 소스 | 대부분 없음. 단, `org.apache.logging.log4j.core` 클래스를 직접 사용하는 커스텀 Appender/Plugin이 있으면 일부 수정 가능 |
| 설정파일 | `log4j2.xml`, `log4j2.properties`, Appender, Layout, Lookup, Filter 설정 검토 필요 |
| pom.xml | `log4j-core`, `log4j-api`, `log4j-slf4j-impl` 버전 통일 필요 |
| 운영환경 | WAS 공통 lib와 애플리케이션 lib 중복 제거, 로그 경로/권한/롤링 정책 검증, SCA 보안점검 필요 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* `log4j-core-2.3.2.jar`를 `2.17.1 이상` 또는 eGovFrame 4.3 검증 버전으로 버전업한다.
* 43번 `log4j-api-2.3.2.jar`, 46번 `log4j-slf4j-impl-2.3.1.jar`와 버전을 동일 계열로 통일한다.
* `log4j-api`만 버전업하지 말고 `log4j-core`를 반드시 함께 교체한다.
* `log4j-to-slf4j`와 `log4j-slf4j-impl` 동시 사용 여부를 확인한다. 두 모듈이 함께 구성되면 로그 순환 위임 문제가 발생할 수 있다.
* `JndiLookup`, `JMSAppender`, `JDBCAppender`, `SocketAppender`, `SMTPAppender` 등 보안·운영 영향이 큰 기능 사용 여부를 점검한다.
* `commons-logging`, `jcl-over-slf4j`, `log4j-over-slf4j`, `slf4j-api`, `logback` 등 로깅 브리지와 구현체 중복 여부를 확인한다.
* WAS 공통 lib와 애플리케이션 `WEB-INF/lib`에 서로 다른 Log4j2 버전이 중복 배치되지 않도록 한다.
* 폐쇄망 반입 시 JAR 파일, 해시값, SBOM, 취약점 점검 결과를 산출물로 관리한다.

### 자동 변환 가능 여부

| 구분 | 가능 여부 |
| --- | --- |
| OpenRewrite | 부분 가능 |
| Python Rule | 가능 |
| LLM 보정 | 필요 |

### 예상 난이도

* 중간

단순 버전업과 JAR 교체만 수행하는 경우 난이도는 낮을 수 있다. 그러나 로그 설정파일이 복잡하거나 커스텀 Appender/Plugin, 비동기 로그, JDBC/JMS Appender, WAS 공통 lib 구성이 포함되어 있으면 난이도는 중간 이상으로 상승한다.

* Log4j2 보안 조치 버전 반영
* `log4j-api`, `log4j-core`, `log4j-slf4j-impl` 버전 정합성 확보
* JNDI, Lookup, JDBC/JMS/Socket Appender 등 보안 영향 설정 점검
* 운영 로그 출력 경로, 파일 롤링 정책, 인코딩, 로그 레벨 검증
* 보안 취약점 점검 결과와 폐쇄망 반입 라이브러리 기준 일치 여부 확인

* SLF4J/Logback 중심으로 로깅 체계를 재정리하는 경우 소스 import 및 설정 변경 가능
* 로그 패턴, Appender 정책, 비동기 로그 구성 변경 시 설정파일 수정 필요

## 7. 위험요소

### 보안

* `log4j-core`는 Log4j2 보안 취약점 점검의 핵심 대상이다.
* 대표적인 Log4j2 취약점으로 CVE-2021-44228, CVE-2021-45046, CVE-2021-45105, CVE-2021-44832 등이 있다.
* Apache 공식 보안 안내에 따르면 Log4Shell 계열 취약점은 `log4j-core`에 영향이 있으며, `log4j-api`만 사용하는 경우는 직접 영향 대상이 아니다.
* `2.3.2`는 일부 취약점에 대한 보안 패치 릴리스로 안내된 이력이 있으나, 현재 전환 기준에서는 구버전 2.3.x 계열을 유지하지 않고 eGovFrame 4.3 검증 버전으로 표준화하는 것이 안전하다.
* 보안점검은 파일명만이 아니라 실제 JAR 내부 클래스, 중복 배포 여부, transitive dependency, WAS 공통 lib까지 포함해 수행해야 한다.

### 운영

* Log4j2 모듈 간 버전이 불일치하면 로그 초기화 실패, 로그 미출력, `NoSuchMethodError`, `ClassNotFoundException`이 발생할 수 있다.
* WAS 공통 lib와 애플리케이션 lib에 서로 다른 Log4j2 버전이 있으면 클래스 로딩 우선순위에 따라 예측하기 어려운 동작이 발생할 수 있다.
* 로그 파일 경로, 디렉터리 권한, 파일 롤링 정책, 보관 기간, 인코딩 변경은 운영 장애로 이어질 수 있다.
* 보안 조치 과정에서 설정을 급하게 변경하면 기존 업무 로그 추적성이 약화될 수 있으므로 로그 포맷 회귀 테스트가 필요하다.

### 호환성

| 항목 | 영향 |
| --- | --- |
| JDK 1.8 | eGovFrame 4.3 기준에서 2.17.1 이상 또는 검증 버전 적용 가능 여부 확인 필요 |
| Spring 5.3 | 일반 로깅 사용 영향 낮음 |
| eGovFrame 4.3 | eGovFrame Lite 4.3 기준 Log4j2 2.23.1 계열 확인됨 |
| SLF4J | `slf4j-api`, `log4j-slf4j-impl`, `jcl-over-slf4j`, `log4j-over-slf4j` 정합성 확인 필요 |
| WAS | 공통 lib 중복 배치 및 클래스 로더 우선순위 확인 필요 |

### 비고

* 43번 `log4j-api-2.3.2.jar`, 44번 `log4j-core-2.3.2.jar`, 46번 `log4j-slf4j-impl-2.3.1.jar`는 하나의 Log4j2 구성으로 묶어 검토해야 한다.
* `log4j-core`는 실제 로그 구현체이므로 43번 `log4j-api`보다 보안·운영 영향도가 높다.
* Matrix에는 Log4Shell 등 보안 취약점 조치 대상으로 정리되어 있으나, Apache 보안 공지 기준 특정 CVE별 영향 버전에는 예외 패치 릴리스가 존재한다. 따라서 보고서·진단 결과에는 “2.3.2가 특정 CVE에 직접 취약”이라고 단정하지 말고 SCA 점검 결과와 Apache 공지 기준을 함께 첨부하는 것이 안전하다.
* eGovFrame Lite 4.3 목록에서는 Log4j2 2.23.1 계열이 확인되므로 프로젝트 표준 라이브러리 목록, 보안검증 기준, 폐쇄망 반입 기준에 맞춰 최종 버전을 확정해야 한다.
* 기존 로그 포맷, 로그 파일명, 로그 레벨, 파일 롤링 정책은 운영 추적성과 장애 대응에 직접 영향을 주므로 회귀 테스트 대상으로 관리해야 한다.

## 참고 근거

* Library Migration Matrix: 44번 `log4j-core-2.3.2.jar`는 `버전업 / 보안` 대상으로 정리되어 있으며, `Log4Shell 등 보안 취약점 조치 대상` 및 `마이그레이션과 별개로 선제 패치 필요`라고 되어 있다.
* Apache Logging Services 보안 공지: Log4Shell 계열 취약점은 `log4j-core`에 영향이 있으며, Log4j2 버전별 보안 조치 기준이 안내되어 있다.
* NVD CVE-2021-44228: Log4j2 JNDI 기능 관련 원격 코드 실행 취약점의 영향 버전과 예외 보안 릴리스가 안내되어 있다.
* eGovFrame Lite 4.3 가이드: `log4j-api 2.23.1`, `log4j-core 2.23.1`, `log4j-slf4j-impl 2.23.1` 항목이 확인된다.
