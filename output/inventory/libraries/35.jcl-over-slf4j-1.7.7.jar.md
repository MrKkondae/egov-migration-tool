## 2. 라이브러리 개요

### 주요 기능

* Apache Commons Logging(JCL) API 호출을 SLF4J API로 전달
* 기존 `org.apache.commons.logging.Log`, `LogFactory` 기반 로그 코드를 유지하면서 로그 구현체를 SLF4J 체계로 통합
* Spring, Apache Commons 계열 등 JCL 기반 라이브러리의 로그를 SLF4J/Logback 또는 SLF4J/Log4j2 체계로 흡수

### 사용 목적

`jcl-over-slf4j`는 Commons Logging API를 사용하는 기존 소스 또는 외부 라이브러리의 로그 호출을 SLF4J로 전달하기 위한 브리지 라이브러리이다.

전자정부프레임워크 3.x 기반 프로젝트에서는 Spring 및 일부 공통 라이브러리에서 Commons Logging API를 사용하는 경우가 많으므로, `commons-logging.jar`를 직접 사용하는 대신 `jcl-over-slf4j.jar`를 배치하여 로그 체계를 SLF4J 중심으로 단일화할 수 있다.

다만 이 라이브러리는 로그 구현체가 아니라 브리지이므로, 실제 로그 출력을 위해서는 `slf4j-api`와 함께 Logback 또는 Log4j2 SLF4J 바인딩이 별도로 구성되어야 한다.

### 대표 사용 예시

```java
import org.apache.commons.logging.Log;
import org.apache.commons.logging.LogFactory;

public class SampleService {
    private static final Log log = LogFactory.getLog(SampleService.class);

    public void execute() {
        log.info("JCL API 호출이 SLF4J로 전달됨");
    }
}
```

위 코드는 소스상으로는 Commons Logging API를 사용하지만, 런타임에 `jcl-over-slf4j.jar`가 적용되면 로그 이벤트가 SLF4J로 전달된다.

## 3. AS-IS 사용 현황

### 프로젝트 내 사용 여부

* 사용 가능성 높음
* 기존 Spring / Commons Logging 기반 라이브러리와 연계 가능
* 실제 사용 여부는 JAR 구성, Maven 의존성, 소스 import 기준으로 확인 필요

### 확인 방법

#### JAR 및 Maven 의존성 확인

```bash
find . -name "*jcl-over-slf4j*.jar"
find . -name "*commons-logging*.jar"
find . -name "*slf4j*.jar"
find . -name "*logback*.jar"
find . -name "*log4j*.jar"

mvn dependency:tree -Dincludes=org.slf4j:jcl-over-slf4j
mvn dependency:tree -Dincludes=commons-logging:commons-logging
mvn dependency:tree -Dincludes=org.slf4j
```

#### 소스 사용처 확인

```bash
grep -R "org.apache.commons.logging" ./src
grep -R "LogFactory.getLog" ./src
grep -R "private static.*Log" ./src
```

#### 설정 파일 확인

```bash
grep -R "commons-logging" .
grep -R "jcl-over-slf4j" .
grep -R "logback" ./src ./config .
grep -R "log4j" ./src ./config .
```

### 예상 사용 위치

* Service

* Spring Framework 및 공통 프레임워크 로그 연계 영역

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상
* 버전업 대상

### 전환 사유

현재 Matrix에서는 `jcl-over-slf4j-1.7.7.jar`를 `jcl-over-slf4j-1.7.36` 또는 eGovFrame 4.3 검증 버전으로 버전업하는 것으로 분류하고 있다.

`jcl-over-slf4j`는 업무 기능 로직 라이브러리가 아니라 로그 브리지 라이브러리이므로 소스 전환 부담은 낮다. 그러나 로그 라이브러리는 클래스패스 구성에 민감하므로 `commons-logging.jar`, `spring-jcl`, `slf4j-api`, `logback-classic`, `log4j-slf4j-impl`, `slf4j-log4j12` 등과의 중복 여부를 반드시 확인해야 한다.

특히 `commons-logging.jar`와 `jcl-over-slf4j.jar`가 동시에 존재하면 동일한 `org.apache.commons.logging` 패키지의 클래스가 중복될 수 있어 예기치 않은 로그 바인딩, 클래스 로딩 충돌, 운영 로그 누락 문제가 발생할 수 있다.

### 영향 범위

| 구분      | 영향 |
| ------- | ---- |
| 소스      | 대부분 없음. 단, `org.apache.commons.logging` 직접 사용 소스는 유지 가능하나 장기적으로 SLF4J 직접 사용 전환 검토 가능 |
| 설정파일    | Logback 또는 Log4j2 설정과 연계 확인 필요 |
| pom.xml | `jcl-over-slf4j` 버전 변경, `commons-logging` exclusion, SLF4J 바인딩 정리 필요 |
| 운영환경    | 로그 출력 여부, 로그 레벨, 로그 파일 경로, WAS 공통 lib 중복 여부 확인 필요 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* `jcl-over-slf4j-1.7.7.jar`를 `jcl-over-slf4j-1.7.36` 또는 eGovFrame 4.3 검증 버전으로 변경
* `slf4j-api`와 동일한 1.7.x 계열로 버전 정합성 확보
* `commons-logging.jar` 중복 제거 또는 Maven exclusion 처리
* Spring 5 환경의 `spring-jcl` 포함 여부 확인
* 실제 로그 구현체를 하나로 통일
  * Logback 사용 시 `logback-classic`, `logback-core` 구성 확인
  * Log4j2 사용 시 `log4j-api`, `log4j-core`, `log4j-slf4j-impl` 버전 통일
* `slf4j-log4j12`, `log4j-to-slf4j`, `log4j-slf4j-impl` 등 상호 충돌 가능한 브리지/바인딩 중복 확인
* 기동 로그에서 SLF4J multiple bindings 경고 여부 확인

### 자동 변환 가능 여부

| 구분          | 가능 여부 |
| ----------- | -------- |
| OpenRewrite | 불필요 / 제한적 |
| Python Rule | 가능 |
| LLM 보정      | 일부 필요 |

### 예상 난이도

* 낮음 ~ 중간

로그 브리지 자체의 버전업은 난이도가 낮지만, 운영 로그 체계 전체를 정리해야 하므로 실제 난이도는 프로젝트의 로그 구성 복잡도에 따라 중간 수준까지 상승할 수 있다.

## 6. 버전 업그레이드 검토

### AS-IS 버전

```text
org.slf4j:jcl-over-slf4j:1.7.7
```

### TO-BE 버전

```text
org.slf4j:jcl-over-slf4j:1.7.36
또는 eGovFrame 4.3 검증 버전
```

### 주요 변경사항

* SLF4J 1.7.x 계열 내 버전업
* Commons Logging API 호출을 SLF4J로 전달하는 기능은 유지
* 업무 소스 API 변경은 원칙적으로 없음
* `slf4j-api`, `jcl-over-slf4j`, 로그 바인딩 라이브러리 간 버전 정합성 확보 필요
* JDK 8 기반 eGovFrame 4.3 환경에서는 SLF4J 2.x로 일괄 전환하기보다 1.7.36 계열 우선 검토가 안정적

### 소스 수정 여부

* 대부분 없음
* 단, 장기적으로 `org.apache.commons.logging` 직접 사용 코드를 `org.slf4j.Logger`로 전환하려는 경우 일부 수정 필요

## 7. 위험요소

### 보안

* `jcl-over-slf4j` 자체는 단순 브리지 라이브러리로 직접적인 보안 위험은 낮음
* 다만 구버전 로그 구현체와 함께 구성될 경우 Log4j 등 다른 로깅 라이브러리 취약점의 영향을 받을 수 있음
* Log4j2 사용 시 `log4j-api`, `log4j-core`, `log4j-slf4j-impl` 보안 패치 버전 통일 필요

### 운영

* `commons-logging.jar`와 `jcl-over-slf4j.jar` 중복 존재 시 클래스 로딩 충돌 가능
* SLF4J 바인딩이 2개 이상 존재하면 다중 바인딩 경고 및 의도하지 않은 로그 출력 가능
* WAS 공통 lib와 애플리케이션 lib에 로그 라이브러리가 중복 배치될 경우 운영과 개발 환경의 로그 동작이 달라질 수 있음
* 로그 경로, 로그 레벨, Rolling 정책 등 운영 로그 설정 회귀 테스트 필요

### 호환성

| 항목            | 영향 |
| --------------- | ---- |
| JDK 8           | 영향 낮음. SLF4J 1.7.36 사용 가능 |
| Spring 5.3      | Spring 자체의 `spring-jcl` 구성과 중복 여부 확인 필요 |
| eGovFrame 4.3   | 로그 체계 정합성 확인 필요 |
| Logback         | SLF4J 바인딩으로 사용 가능 |
| Log4j2          | `log4j-slf4j-impl`과 버전 통일 필요 |
| Commons Logging | 원본 `commons-logging.jar` 제거 또는 exclusion 필요 |

### 비고

* 본 라이브러리는 업무 기능 라이브러리가 아니라 로그 브리지 라이브러리이므로 업무 소스 영향은 낮다.
* 그러나 로그 라이브러리 중복 구성은 운영 장애 분석과 감사 로그 확인에 직접 영향을 줄 수 있으므로 운영 검증 중요도는 높다.
* `commons-logging.jar`와 `jcl-over-slf4j.jar`는 동시에 유지하지 않는 것을 원칙으로 한다.
* Spring 5 기반 eGovFrame 4.3 환경에서는 `spring-jcl` 존재 여부도 함께 확인해야 한다.
* 45번 `log4j-over-slf4j`, 46번 `log4j-slf4j-impl` 항목과 함께 전체 로깅 체계 기준을 수립하는 것이 바람직하다.
