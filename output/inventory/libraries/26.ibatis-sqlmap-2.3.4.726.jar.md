## 2. 라이브러리 개요

### 주요 기능

* SQL 문을 Java 소스에서 분리하여 XML 기반 SQL Map 파일로 관리
* JavaBean, Map, 기본 타입과 SQL 파라미터 및 조회 결과 매핑
* JDBC 코드의 반복 처리, 파라미터 바인딩, 결과 매핑, 예외 처리 단순화
* Spring 연계 시 `SqlMapClient`, `SqlMapClientTemplate` 기반 DAO 구현 지원

### 사용 목적

iBATIS SQL Maps는 Java 애플리케이션의 데이터 접근 계층에서 SQL과 Java 객체 간의 매핑을 지원하는 Data Mapper 프레임워크이다. 전자정부프레임워크 구버전에서는 JDBC 기반 Data Access 처리를 추상화하기 위한 기반 오픈소스로 iBATIS를 채택하여 사용한 이력이 있다.

다만 Apache iBATIS 프로젝트는 2010년 Apache Software Foundation에서 retired 처리되었고, 원 프로젝트 팀은 MyBatis 프로젝트로 이동하였다. 따라서 eGovFrame 3.1에서 4.3으로 전환하는 경우 `ibatis-sqlmap-2.3.4.726.jar`는 유지가 아니라 MyBatis 기반 구조로 대체하는 것이 타당하다.

### 대표 사용 예시

#### AS-IS: iBATIS Java DAO 예시

```java
import com.ibatis.sqlmap.client.SqlMapClient;
import org.springframework.orm.ibatis.SqlMapClientTemplate;

public class SampleDao {
    private SqlMapClientTemplate sqlMapClientTemplate;

    public Object selectSample(Object param) {
        return sqlMapClientTemplate.queryForObject("sample.selectSample", param);
    }

    public List<?> selectSampleList(Object param) {
        return sqlMapClientTemplate.queryForList("sample.selectSampleList", param);
    }
}
```

#### AS-IS: iBATIS SQL Map XML 예시

```xml
<sqlMap namespace="sample">
    <select id="selectSample"
            parameterClass="sample.vo.SampleVO"
            resultClass="sample.vo.SampleVO">
        SELECT ID, NAME
          FROM SAMPLE
         WHERE ID = #id#
    </select>
</sqlMap>
```

#### TO-BE: MyBatis Mapper XML 예시

```xml
<mapper namespace="sample.mapper.SampleMapper">
    <select id="selectSample"
            parameterType="sample.vo.SampleVO"
            resultType="sample.vo.SampleVO">
        SELECT ID, NAME
          FROM SAMPLE
         WHERE ID = #{id}
    </select>
</mapper>
```

#### TO-BE: MyBatis Mapper Interface 예시

```java
import org.egovframe.rte.psl.dataaccess.mapper.Mapper;

@Mapper("sampleMapper")
public interface SampleMapper {
    SampleVO selectSample(SampleVO param);
}
```

```bash
grep -R "com.ibatis" ./src

grep -R "SqlMapClientTemplate" ./src
```

```bash
grep -R "SqlMapClientFactoryBean" ./src ./config .
grep -R "sqlMapClient" ./src ./config .

```

#### SQL Map XML 확인

```bash
grep -R "<sqlMap" ./src ./config .
grep -R "parameterClass\|resultClass\|resultMap\|cacheModel" ./src ./config .

grep -R "#[A-Za-z0-9_.]*#" ./src ./config .
find . -iname "*sqlmap*.xml" -o -iname "*sql-map*.xml" -o -iname "*SqlMap*.xml"
```

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상
* 조치 유형: 대체
* 권장 전환 방향: `org.egovframe.rte:org.egovframe.rte.psl.dataaccess:4.3.0` 기반 MyBatis 적용 또는 MyBatis 3.5.x + mybatis-spring 2.1.x 계열 적용

### 전환 사유

Apache iBATIS는 2010년에 retired 처리된 프로젝트이며, 현재 유지보수되는 계열은 MyBatis이다. eGovFrame 4.3 실행환경은 Spring Framework 5.3.37, JDK 1.8 이상, MyBatis 3.5.x 계열을 기준으로 구성되므로 iBATIS 2.x를 유지하는 것은 장기 유지보수, 보안 점검, 표준 프레임워크 정합성 측면에서 부적합하다.

또한 iBATIS 2.x와 MyBatis 3.x는 패키지명, 설정 방식, Mapper XML 구조, 동적 SQL 태그가 다르므로 단순 JAR 교체만으로 전환할 수 없다. DAO 소스, SQL Map XML, Spring 설정을 함께 전환해야 한다.

### 영향 범위

| 구분 | 영향 |
| --- | --- |
| 소스 | 높음. `com.ibatis.*`, `SqlMapClient`, `SqlMapClientTemplate`, `EgovAbstractDAO` 사용 DAO 수정 필요 |
| 설정파일 | 높음. `SqlMapClientFactoryBean`, `sqlMapClient`, `sql-map-config.xml`, SQL Map XML 등록 방식 변경 필요 |
| pom.xml | 높음. `ibatis-sqlmap` 제거, `org.egovframe.rte.psl.dataaccess` 4.3.0 또는 MyBatis/MyBatis-Spring 의존성 적용 필요 |
| 운영환경 | 중간~높음. DB 접속 자체보다 SQL 실행 결과, 트랜잭션, 동적 SQL, 페이징/목록 조회 결과 회귀 테스트 필요 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* `ibatis-sqlmap-2.3.4.726.jar` 제거
* `org.egovframe.rte:org.egovframe.rte.psl.dataaccess:4.3.0` 또는 프로젝트 표준에 맞는 MyBatis 3.5.x / mybatis-spring 2.1.x 적용
* Java DAO의 iBATIS API 제거
  * `com.ibatis.sqlmap.client.SqlMapClient` 제거
  * `org.springframework.orm.ibatis.SqlMapClientTemplate` 제거
  * `queryForObject`, `queryForList` 호출부를 `selectOne`, `selectList` 또는 Mapper Interface 호출 방식으로 변경
* Spring 설정 변경
  * `SqlMapClientFactoryBean` → `SqlSessionFactoryBean`
  * `SqlMapClientTemplate` → `SqlSessionTemplate`
  * SQL Map 목록 등록 → MyBatis mapper XML 위치 등록
  * Mapper Interface 사용 시 `MapperConfigurer` 또는 Mapper Scan 설정 추가
* XML SQL Map 변경
  * `<sqlMap>` → `<mapper>`
  * `parameterClass` → `parameterType`
  * `resultClass` → `resultType`
  * `#param#` → `#{param}`
  * iBATIS 동적 SQL 태그를 MyBatis 동적 SQL 태그로 재작성
* 트랜잭션 설정 확인
  * 기존 Spring 트랜잭션 경계 유지 여부 확인
  * `DataSourceTransactionManager` 또는 WAS/JTA 트랜잭션 연계 여부 확인
* SQL 실행 결과 회귀 테스트
  * 단건 조회, 목록 조회, 등록, 수정, 삭제
  * 동적 조건 검색
  * null/empty 조건 처리
  * 페이징 처리
  * 대량 조회 및 배치 처리

### 자동 변환 가능 여부

| 구분 | 가능 여부 |
| --- | --- |
| OpenRewrite | 부분 가능. 의존성 탐지, import 탐지, 일부 Java API 치환은 Custom Recipe로 가능하나 SQL XML 의미 변환은 제한적 |
| Python Rule | 부분 가능. `parameterClass`/`resultClass`/`#param#` 등 단순 패턴 치환과 사용처 목록화는 가능하나 동적 SQL 의미 검증은 수동 검토 필요 |
| LLM 보정 | 필요. 동적 SQL, DAO 호출 구조, namespace/id 정합성, Mapper Interface 설계 보정 필요 |

### 예상 난이도

* 높음

### 우선 점검 대상 패턴

| AS-IS 패턴 | TO-BE 검토 방향 |
| --- | --- |
| `com.ibatis.sqlmap.client.SqlMapClient` | `SqlSessionFactory`, `SqlSessionTemplate`, Mapper Interface로 대체 |
| `SqlMapClientTemplate.queryForObject` | `SqlSessionTemplate.selectOne` 또는 Mapper 메서드 호출 |
| `SqlMapClientTemplate.queryForList` | `SqlSessionTemplate.selectList` 또는 Mapper 메서드 호출 |
| `<sqlMap namespace="...">` | `<mapper namespace="...">` |
| `parameterClass` | `parameterType` |
| `resultClass` | `resultType` |
| `#param#` | `#{param}` |
| `<isNotEmpty>`, `<isEqual>`, `<iterate>` | `<if>`, `<choose>`, `<foreach>`, `<where>`, `<trim>` |
| `sql-map-config.xml` | `mybatis-config.xml` 또는 Spring mapperLocations 설정 |

## 6. 버전 업그레이드 검토

### AS-IS 버전

```text
org.apache.ibatis:ibatis-sqlmap:2.3.4.726
```

### TO-BE 버전

```text
권장 1안:
org.egovframe.rte:org.egovframe.rte.psl.dataaccess:4.3.0

권장 2안:
org.mybatis:mybatis:3.5.x
org.mybatis:mybatis-spring:2.1.x
```

### 주요 변경사항

* Apache iBATIS 2.x에서 MyBatis 3.x로 대체
* iBATIS API 기반 DAO에서 MyBatis Mapper 기반 DAO로 변경
* SQL Map XML 구조 변경
* 동적 SQL 태그 체계 변경
* Spring iBATIS 연계 설정 제거 및 MyBatis-Spring 연계 설정 적용
* eGovFrame 4.x 계열 패키지 기준에 따라 `egovframework.rte.*`에서 `org.egovframe.rte.*` 계열로 전환 검토 필요

### 소스 수정 여부

* 대규모 수정 필요

단순 JAR 교체 대상이 아니며, DAO 소스와 SQL XML을 함께 전환해야 한다. 특히 SQL Map XML 수가 많거나 공통 DAO 기반 호출 구조가 복잡한 경우 자동 치환 후 업무별 회귀 테스트가 필수이다.

## 7. 위험요소

### 보안

* iBATIS 자체가 retired 상태이므로 신규 보안 패치 또는 유지보수 대응을 기대하기 어렵다.
* `ibatis-sqlmap` 클래스가 Jackson Databind 취약점의 gadget 관련 요소로 언급된 사례가 있으므로, 단순히 직접 취약점이 없다고 판단하기보다는 클래스패스 노출 관점에서 제거하는 것이 안전하다.
* Jackson 1.x/2.x 구버전과 함께 사용 중인 경우 역직렬화 취약점 점검을 병행해야 한다.

### 운영

* 동적 SQL 변환 오류 시 조회 조건 누락, 과다 조회, 결과 건수 차이, 성능 저하가 발생할 수 있다.
* `null`, 빈 문자열, 숫자 0, 컬렉션 empty 여부에 대한 iBATIS/MyBatis 조건 처리 차이를 확인해야 한다.
* 기존 SQL Map namespace와 statement id를 Mapper namespace/id로 재정의할 때 호출부와 XML 간 불일치가 발생할 수 있다.
* 대량 목록, 배치, 페이징, 트랜잭션 롤백 시나리오 회귀 테스트가 필요하다.

### 호환성

| 항목 | 영향 |
| --- | --- |
| JDK 8 이상 | eGovFrame 4.3 기준 JDK 1.8 이상 필요. iBATIS 자체보다 eGovFrame/Spring/MyBatis 조합 기준으로 검증 필요 |
| Spring 5.3.x | Spring iBATIS 지원 구조와 맞지 않으므로 MyBatis-Spring 2.1.x 계열 또는 eGovFrame DataAccess 사용 필요 |
| eGovFrame 4.3 | `org.egovframe.rte.psl.dataaccess` 기반 MyBatis 구성과 정합성 확보 필요 |
| DBMS | SQL 자체는 대부분 유지 가능하나 동적 SQL, 파라미터 바인딩, JDBC 타입 지정 차이 확인 필요 |
