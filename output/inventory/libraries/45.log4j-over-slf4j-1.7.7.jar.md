## 2. 라이브러리 개요

### 주요 기능

* Log4j 1.x API 호출을 SLF4J API로 전달
* 기존 `org.apache.log4j.Logger` 기반 소스의 호환성 유지
* Log4j 1.x 구현체(`log4j.jar`) 제거 후 SLF4J 기반 로깅 체계로 통합 지원
* Logback 또는 Log4j2 등 실제 로깅 구현체와 연계

### 사용 목적

`log4j-over-slf4j`는 기존 소스나 외부 라이브러리가 Log4j 1.x API를 직접 호출하는 경우, 소스 수정 없이 해당 호출을 SLF4J로 전달하기 위한 브리지 라이브러리이다.

즉, 다음과 같은 기존 코드가 있을 때 실제 로그 처리는 SLF4J 바인딩과 하위 로깅 구현체가 담당하도록 연결한다.

```java
import org.apache.log4j.Logger;

public class SampleService {
    private static final Logger LOGGER = Logger.getLogger(SampleService.class);

    public void execute() {
        LOGGER.info("Log4j 1.x API 호출");
    }
}
```

전자정부프레임워크 4.3 전환 시에는 Log4j 1.x 직접 사용을 줄이고 SLF4J 기반 로깅 구조로 정리하는 것이 바람직하다. 다만 업무 소스 또는 외부 라이브러리에 `org.apache.log4j.*` 사용이 남아 있으면 `log4j-over-slf4j`가 필요할 수 있다.

### 대표 사용 예시

#### AS-IS 코드

```java
import org.apache.log4j.Logger;

public class LegacyLoggerSample {
    private static final Logger LOGGER = Logger.getLogger(LegacyLoggerSample.class);

    public void test() {
        LOGGER.info("legacy log4j logging");
    }
}
```

#### TO-BE 권장 코드

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class Slf4jLoggerSample {
    private static final Logger LOGGER = LoggerFactory.getLogger(Slf4jLoggerSample.class);

    public void test() {
        LOGGER.info("slf4j logging");
    }
}
```

#### 브리지 적용 시

```xml
<dependency>
    <groupId>org.slf4j</groupId>
    <artifactId>log4j-over-slf4j</artifactId>
    <version>1.7.36</version>
</dependency>
```

## 3. AS-IS 사용 현황

### 프로젝트 내 사용 여부

* 확인 필요
* 기존 Log4j 1.x API 직접 사용 시 필요
* 직접 사용이 없고 외부 라이브러리도 Log4j 1.x API를 요구하지 않으면 제거 가능

### 확인 방법

#### JAR 존재 여부 확인

```bash
find . -name "log4j-over-slf4j*.jar"
find . -name "log4j*.jar"
find . -name "slf4j*.jar"
```

#### 소스 내 Log4j 1.x API 사용 여부 확인

```bash
grep -R "org.apache.log4j" ./src
grep -R "Logger.getLogger" ./src
grep -R "PropertyConfigurator" ./src
grep -R "DOMConfigurator" ./src
grep -R "MDC" ./src
grep -R "NDC" ./src
```

#### Maven 의존성 확인

```bash
mvn dependency:tree -Dincludes=org.slf4j:log4j-over-slf4j
mvn dependency:tree -Dincludes=log4j:log4j
mvn dependency:tree -Dincludes=org.slf4j:slf4j-log4j12
mvn dependency:tree -Dincludes=org.slf4j:slf4j-reload4j
mvn dependency:tree -Dincludes=org.apache.logging.log4j:log4j-slf4j-impl
```

#### 중복/충돌 라이브러리 확인

```bash
find ./WEB-INF/lib -name "*log4j*.jar"
find ./WEB-INF/lib -name "*slf4j*.jar"
find ./WEB-INF/lib -name "*logback*.jar"
```

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상 / 버전업 / 검토

### 전환 사유

`log4j-over-slf4j-1.7.7.jar`는 SLF4J 1.7.x 초기 계열로 오래된 버전이다.

전자정부프레임워크 4.3 전환 시 로깅 체계는 다음 중 하나로 명확히 정리해야 한다.

* SLF4J + Logback
* SLF4J + Log4j2
* eGovFrame 4.3 검증 라이브러리 목록 기준 구성

이 라이브러리는 Log4j 1.x API 호환을 위한 브리지이므로, 실제 소스에서 `org.apache.log4j.*`를 직접 사용하면 유지하되 `1.7.36` 또는 eGovFrame 4.3 검증 버전으로 버전업하는 것이 적절하다.

반대로 직접 사용이 없고 외부 라이브러리에서도 Log4j 1.x API를 요구하지 않으면 제거할 수 있다.

### 영향 범위

| 구분      | 영향 |
| ------- | ---- |
| 소스      | `org.apache.log4j.*` 직접 사용 시 영향 있음. SLF4J 표준화 시 import 변경 필요 |
| 설정파일    | `log4j.properties`, `log4j.xml`, `log4j2.xml`, `logback.xml` 구성 정리 필요 |
| pom.xml | `log4j-over-slf4j`, `slf4j-api`, 로깅 구현체 버전 정합성 필요 |
| 운영환경    | WAS 공통 lib와 애플리케이션 lib 중복 시 클래스 충돌 가능 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* `log4j-over-slf4j-1.7.7.jar`를 `1.7.36` 또는 eGovFrame 4.3 검증 버전으로 버전업한다.
* `log4j:log4j` 즉 Log4j 1.x 구현체가 함께 포함되어 있으면 제거한다.
* `slf4j-log4j12.jar` 또는 `slf4j-reload4j.jar`와 동시 구성하지 않는다.
* `log4j-to-slf4j`와 `log4j-slf4j-impl`처럼 상호 브리징 구조가 만들어지지 않도록 점검한다.
* `org.apache.log4j.Logger` 직접 사용 소스는 장기적으로 `org.slf4j.Logger`로 전환한다.
* 실제 로그 구현체를 Logback 또는 Log4j2 중 하나로 확정한다.
* WAS 공통 lib와 `WEB-INF/lib`에 로깅 라이브러리가 중복 배치되어 있는지 확인한다.

### 자동 변환 가능 여부

| 구분          | 가능 여부 |
| ----------- | -------- |
| OpenRewrite | 부분 가능 |
| Python Rule | 가능 |
| LLM 보정      | 필요 |

### 예상 난이도

* 중간

단순 버전업은 난이도가 낮지만, 운영환경에 여러 로깅 브리지와 구현체가 섞여 있으면 기동 오류, 로그 미출력, 이중 출력, 무한 위임 문제가 발생할 수 있어 검토 난이도는 중간으로 본다.


### 주요 변경사항

* SLF4J 1.7.x 계열 내 안정 버전으로 버전업
* Log4j 1.x API 호환 브리지 역할은 유지
* `slf4j-api` 및 실제 로깅 구현체와 버전 정합성 확보 필요
* 보안 취약점이 많은 Log4j 1.x 구현체를 직접 유지하지 않고 브리지 방식으로 분리 가능
* SLF4J 2.x 계열로 전환하는 경우 바인딩 방식이 변경되므로 별도 검토 필요

### 소스 수정 여부

* 단순 버전업: 대부분 없음
* SLF4J 표준화: 일부 수정 필요
* Log4j 1.x 설정 API 직접 사용: 일부 수정 필요
* 커스텀 Appender 또는 `PropertyConfigurator`, `DOMConfigurator` 직접 사용: 수동 검토 필요

## 7. 위험요소

### 보안

* `log4j-over-slf4j` 자체는 Log4j 1.x 구현체가 아니라 브리지 라이브러리이다.
* 다만 `log4j-1.2.x.jar`가 함께 포함되어 있으면 Log4j 1.x 보안 이슈와 운영 리스크가 남을 수 있다.
* Log4j2 사용 시에는 `log4j-api`, `log4j-core`, `log4j-slf4j-impl` 버전을 동일 계열로 통일해야 한다.
* 보안점검은 파일명만 보지 말고 실제 포함 JAR 전체를 기준으로 수행해야 한다.

### 운영

* `log4j-over-slf4j.jar`와 `slf4j-log4j12.jar`를 함께 배치하면 로깅 호출이 순환될 수 있다.
* 여러 SLF4J Binding이 동시에 존재하면 기동 시 경고 또는 의도하지 않은 구현체 선택이 발생할 수 있다.
* WAS 공통 lib에 구버전 SLF4J 또는 Log4j가 있으면 애플리케이션 버전업 효과가 반영되지 않을 수 있다.
* 로그 포맷, 파일 경로, 롤링 정책, 로그 레벨이 기존 운영환경과 달라질 수 있다.

### 호환성

| 항목              | 영향 |
| ----------------- | ---- |
| JDK 1.8           | SLF4J 1.7.36 사용 가능 |
| Spring 5.3        | 직접 영향 낮음 |
| eGovFrame 4.3     | SLF4J 기반 로깅 체계와 정합성 확인 필요 |
| Log4j 1.x API     | 브리지로 호환 가능 |
| Log4j2            | `log4j-slf4j-impl` 구성 시 버전 통일 필요 |
| Logback           | `logback-classic`, `logback-core`와 함께 구성 가능 |

```text
전환대상여부 : 대상 / 버전업 / 검토

조치방안 :
- log4j-over-slf4j-1.7.7.jar를 1.7.36 또는 eGovFrame 4.3 검증 버전으로 버전업
- 소스에서 org.apache.log4j.Logger를 직접 사용하면 유지
- 직접 사용이 없고 외부 라이브러리 의존도 없으면 제거 검토
- log4j-1.x 구현체, slf4j-log4j12, slf4j-reload4j와의 중복/충돌 여부 확인
- 최종 로그 구현체를 Logback 또는 Log4j2 중 하나로 확정
- 기동 로그, 업무 로그, 배치 로그, 오류 로그 출력 회귀 테스트 수행

자동변환 :
- 부분 가능

소스수정 :
- 대부분 없음
- SLF4J 표준화 시 일부 수정 필요

위험도 :
- 중간
```

### 비고

* 본 라이브러리는 Log4j 1.x 구현체가 아니라 Log4j 1.x API를 SLF4J로 전달하는 브리지이다.
* `log4j-over-slf4j.jar`를 사용하려면 실제 로그를 출력할 SLF4J 바인딩과 구현체가 별도로 필요하다.
* `log4j-over-slf4j.jar`와 `slf4j-log4j12.jar` 또는 `slf4j-reload4j.jar`를 동시에 배치하지 않도록 한다.
* 기존 `log4j.properties` 또는 `log4j.xml` 설정 파일이 실제로 사용되는지 확인하고, Logback 또는 Log4j2 설정으로 정리한다.
* 장기적으로는 `org.apache.log4j.*` 직접 사용 코드를 `org.slf4j.*`로 전환하는 것이 바람직하다.
