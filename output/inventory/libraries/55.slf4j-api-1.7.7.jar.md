## 2. 라이브러리 개요

### 주요 기능

* Java 로깅 추상화 API 제공
* Log4j, Logback, JUL 등 로깅 구현체와 연계
* `Logger`, `LoggerFactory` 기반 로깅 처리
* `{}` placeholder 기반 파라미터 로깅 지원
* 애플리케이션 로그 호출부와 실제 로깅 구현체 분리

### 사용 목적

SLF4J는 애플리케이션 코드에서 특정 로깅 구현체에 직접 의존하지 않도록 하는 로깅 추상화 라이브러리이다.
전자정부프레임워크에서도 SLF4J API를 추가하고, 실제 로깅 구현체는 별도 Binding Jar로 연결하는 구조를 설명하고 있다. ([eGovFrame][1])

### 대표 사용 예시

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class SampleService {

    private static final Logger LOGGER =
        LoggerFactory.getLogger(SampleService.class);

    public void execute() {
        LOGGER.debug("SLF4J Logger - {}", "execute");
    }
}
```

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상

### 전환 사유

`slf4j-api-1.7.7.jar`는 오래된 SLF4J 1.7 계열 라이브러리이다. Maven Repository 기준 `slf4j-api 1.7.7`은 약 12년 전 배포된 버전으로 확인된다. ([Maven Repository][2])

eGovFrame 4.3 Lite 라이브러리 목록에는 `jcl-over-slf4j 2.0.13`, `log4j-over-slf4j 2.0.13`, `log4j-api 2.23.1`, `log4j-core 2.23.1`, `log4j-slf4j-impl 2.23.1` 등이 포함되어 있어 로깅 계열 라이브러리 정합성 검토가 필요하다. ([eGovFrame][3])

따라서 단순히 `slf4j-api`만 교체하는 것이 아니라, Log4j2, Logback, Bridge Jar, Binding Jar 전체를 함께 점검해야 한다.

### 영향 범위

| 구분      | 영향 |
| ------- | -- |
| 소스      | 낮음 |
| 설정파일    | 중간 |
| pom.xml | 높음 |
| 운영환경    | 중간 |

##### (13) 최종 정리

```text
slf4j-api 1.7.7 → 1.7.36 전환은 Java 소스 수정 영향이 거의 없다.

기존 org.slf4j.Logger, LoggerFactory, LOGGER.debug/info/warn/error 사용 코드는
대부분 그대로 유지 가능하다.

OpenRewrite를 통해 Maven 의존성 버전 변경과 구버전 Jar 탐지는 자동화할 수 있다.

다만 slf4j-api는 로그 API만 제공하므로 실제 로그 출력 여부는
Binding Jar(logback-classic, log4j-slf4j-impl, slf4j-log4j12 등)와
Bridge Jar(jcl-over-slf4j, log4j-over-slf4j 등)의 정합성에 따라 결정된다.

따라서 자동변환 대상은 slf4j-api 버전 변경과 중복 Binding 탐지까지로 보고,
Binding 제거, Log4j 1.x 대체, 실제 로그 출력 검증은 수동 검토 대상으로 관리한다.
```
