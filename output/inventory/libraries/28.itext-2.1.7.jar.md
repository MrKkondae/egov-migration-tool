## 2. 라이브러리 개요

### 주요 기능

* PDF 문서 생성
* PDF 문단, 표, 이미지, 폰트, 페이지 구성 처리
* 기존 PDF 읽기, 병합, 일부 수정 처리
* PDF 암호화, 권한 설정, 메타데이터 처리
* 출력 스트림 기반 PDF 다운로드 처리

### 사용 목적

iText는 Java 애플리케이션에서 PDF 파일을 생성하거나 수정하기 위해 사용하는 PDF 처리 라이브러리이다.
전자정부프레임워크 3.x 기반 시스템에서는 증명서, 신청서, 출력물, 통계표, 배치성 PDF 생성, 전자서명 대상 PDF 생성 전처리 등에 사용되었을 가능성이 있다.

AS-IS의 `itext-2.1.7.jar`는 `com.lowagie` 패키지 기반의 오래된 iText 2.x 계열이다. 최신 iText 계열은 라이선스 및 API 구조가 달라졌으므로 단순 버전업보다는 대체 전환으로 보는 것이 적절하다.

본 항목은 `Library Migration Matrix` 기준으로 `OpenPDF 1.3.x` 또는 상용 `iText 7` 대체 대상으로 분류한다.

### 대표 사용 예시

```java
import java.io.FileOutputStream;

import com.lowagie.text.Document;
import com.lowagie.text.Paragraph;
import com.lowagie.text.pdf.PdfWriter;

public class PdfCreateSample {
    public void createPdf(String filePath) throws Exception {
        Document document = new Document();
        PdfWriter.getInstance(document, new FileOutputStream(filePath));

        document.open();
        document.add(new Paragraph("PDF 생성 예시"));
        document.close();
    }
}
```

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상

### 전환 사유

`itext-2.1.7.jar`는 오래된 PDF 라이브러리이며, eGovFrame 4.3 전환 시 라이브러리 최신화, 라이선스 검토, 보안 점검, JDK 1.8 호환성 검증이 필요하다.

iText 5 이상은 AGPL 또는 상용 라이선스 모델로 운영되므로, 공공기관 시스템에서 무상 오픈소스 형태로 단순 버전업하는 경우 라이선스 의무사항 검토가 필요하다. 따라서 기존 `com.lowagie` 패키지 기반 소스 변경 부담을 줄이려면 OpenPDF의 JDK 8 호환 계열을 우선 검토하고, 고급 PDF 기능이나 상용 지원이 필요한 경우 iText 7 이상 상용 라이선스를 별도 검토하는 방향이 적절하다.

### 영향 범위

| 구분      | 영향 |
| ------- | ---- |
| 소스      | `com.lowagie.text.*`, `PdfWriter`, `PdfReader`, `PdfStamper`, `PdfPTable` 등 사용 소스 영향 가능 |
| 설정파일    | 직접 설정 영향은 낮으나 폰트 경로, PDF 템플릿 경로, 다운로드 MIME 설정, 전자서명 연계 설정 확인 필요 |
| pom.xml | `com.lowagie:itext:2.1.7` 또는 로컬 JAR 제거 후 OpenPDF 또는 상용 iText 의존성으로 변경 필요 |
| 운영환경    | 한글 폰트, CJK 인코딩, 파일 시스템 경로, PDF 생성 권한, WAS 메모리 사용량, 전자서명/암호화 연계 영향 가능 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* `itext-2.1.7.jar` 직접 포함 여부 확인
* 소스 내 `com.lowagie.*` import 사용 현황 목록화
* PDF 생성 기능별 유형 분류
  * 단순 PDF 생성
  * 표/이미지/폰트 포함 PDF 생성
  * 기존 PDF 수정 및 병합
  * PDF 암호화
  * PDF 전자서명 연계
  * 리포트 솔루션 또는 별도 출력 모듈 연계
* 대체 라이브러리 결정
  * 1차 후보: OpenPDF JDK 8 호환 계열
  * 2차 후보: 상용 iText 7 이상
  * 기능 성격에 따라 Apache PDFBox 병행 검토 가능
* 한글 폰트, 줄바꿈, 표 레이아웃, 페이지 나눔, 이미지 품질 회귀 테스트
* PDF 전자서명, 타임스탬프, 인증서 모듈과 연계되는 경우 공급사 API 호환성 확인
* 라이선스 검토 결과를 오픈소스 검증 산출물에 포함

### 자동 변환 가능 여부

| 구분          | 가능 여부 |
| ----------- | -------- |
| OpenRewrite | 제한적 가능. Maven 의존성 변경, 일부 import 변경은 가능하나 PDF 생성 로직 의미 변환은 어려움 |
| Python Rule | 부분 가능. `com.lowagie` 사용처 목록화, 단순 import 치환, PDF 관련 클래스 사용 빈도 집계 가능 |
| LLM 보정      | 필요. 표/폰트/전자서명/암호화/기존 PDF 수정 로직은 수동 검토와 LLM 보정 필요 |

### 예상 난이도

* 중간
* 단, PDF 전자서명, 암호화, 기존 PDF 수정, 정교한 출력 서식이 포함된 경우 높음

## 6. 버전 업그레이드 검토

### AS-IS 버전

```text
com.lowagie:itext:2.1.7
```

### TO-BE 버전

```text
1차 권장 후보:
com.github.librepdf:openpdf:1.3.x 계열 또는 프로젝트 표준으로 검증된 JDK 8 호환 OpenPDF 버전

대안 후보:
상용 라이선스 기반 iText 7 이상
```

### 주요 변경사항

* iText 2.1.7은 `com.lowagie` 패키지를 사용한다.
* OpenPDF는 iText 계열의 오픈소스 후속/포크 성격으로 PDF 생성·편집 기능을 제공한다.
* OpenPDF의 최신 계열은 Java 요구사항 및 패키지 구조가 변경되고 있으므로, eGovFrame 4.3의 JDK 1.8 기준에서는 최신 버전 일괄 적용보다 JDK 8 호환 검증 버전을 우선 적용한다.
* iText 5 이상은 AGPL 또는 상용 라이선스 모델이므로, 폐쇄형 업무시스템에서 무상 AGPL 사용 시 라이선스 의무사항을 법무 또는 오픈소스 검증 절차로 확인해야 한다.
* iText 7 이상은 API 구조가 `com.itextpdf.*` 계열로 크게 변경되어 단순 치환 전환이 어렵다.
* OpenPDF 호환 계열 적용 시 기존 `com.lowagie.text.*` import를 상당 부분 유지할 수 있으나, 모든 PDF 기능이 무조건 동일하게 동작한다고 전제하면 안 된다.

### 소스 수정 여부

* OpenPDF 호환 계열 적용 시: 일부 수정 필요
* iText 7 이상 적용 시: 대규모 수정 필요
* PDF 전자서명/암호화/템플릿 수정 기능 포함 시: 대규모 수정 가능성 있음

## 7. 위험요소

### 보안

* iText 2.1.7은 오래된 라이브러리이므로 최신 보안 점검 도구 기준의 취약점 여부 확인 필요
* PDF 파일을 외부 입력으로 읽거나 병합하는 경우 악성 PDF, 대용량 PDF, 압축 폭탄, 메모리 사용량 증가 위험 검토 필요
* OpenPDF 공식 문서에서도 입력값 신뢰성, 검증, 샌드박싱은 애플리케이션 개발자 책임임을 안내하고 있으므로 PDF 입력 데이터에 대한 검증 로직 필요
* 전자서명 또는 암호화 기능을 사용하는 경우 BouncyCastle 등 암호화 라이브러리와의 버전 충돌 확인 필요

### 운영

* PDF 출력물은 사용자가 눈으로 확인하는 산출물이므로 미세한 폰트, 줄바꿈, 페이지 나눔 차이도 장애 또는 민원으로 이어질 수 있음
* 한글 폰트 파일 경로, 운영서버 폰트 설치 여부, WAS 권한, 임시파일 경로 확인 필요
* 대량 PDF 생성 배치의 경우 CPU, 메모리, 디스크 I/O 영향 확인 필요
* PDF 전자서명 모듈, TSA 모듈, 리포팅툴과 연계된 경우 단독 라이브러리 전환만으로 판단하면 안 됨

### 호환성

| 항목            | 영향 |
| --------------- | ---- |
| JDK 1.8         | eGovFrame 4.3 기본 요구사항이므로 OpenPDF 버전 선정 시 JDK 8 호환성 확인 필요 |
| Spring 5.3.37   | PDF 라이브러리 자체 영향은 낮으나 Controller 다운로드 응답 처리와 예외 처리 확인 필요 |
| eGovFrame 4.3   | 직접 포함 라이브러리는 아니므로 프레임워크 핵심 영향은 낮지만, 업무 PDF 기능의 회귀 테스트 필요 |
| WAS             | 폰트, 파일 경로, classloader 중복 JAR, 메모리 사용량 확인 필요 |
| 전자서명 모듈    | PDF 생성 결과물 구조 변경 시 서명 검증 실패 가능성 있음 |

### 비고

* `Library Migration Matrix`에는 28번 항목이 `itext-2.1.7.jar`이며, `OpenPDF 1.3.x` 또는 상용 `iText 7` 대체로 정리되어 있다.
* 공공기관 사업에서는 iText 5 이상 AGPL 라이선스 적용 여부를 반드시 검토해야 하며, AGPL 의무사항을 수용하기 어렵다면 OpenPDF 또는 상용 라이선스 도입을 우선 검토한다.
* OpenPDF 최신 계열은 Java 요구사항 및 패키지 구조가 변경되고 있으므로, eGovFrame 4.3의 JDK 1.8 기준에 맞는 버전을 별도 검증해야 한다.
* PDF 출력 결과는 기능 테스트뿐 아니라 눈검수 기준의 레이아웃 비교가 필요하다.
* PDF 전자서명, TSA, 인증서 모듈과 연계되는 경우 관련 공급사 라이브러리와 통합 테스트가 필요하다.
* `Library Migration Matrix.md` 파일에는 Git 병합 충돌 마커가 남아 있으므로 최종 산출물 반영 전 Matrix 정리가 필요하다.
