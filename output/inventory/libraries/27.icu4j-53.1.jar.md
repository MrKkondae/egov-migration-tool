## 2. 라이브러리 개요

### 주요 기능

* Unicode 문자열 처리 및 문자 속성 처리
* Unicode 정규화(NFC, NFD, NFKC, NFKD)
* 다국어 문자열 정렬 및 비교(Collation)
* 로케일 기반 날짜, 시간, 숫자, 통화 포맷 처리
* 문자 변환, 대소문자 변환, 전각/반각 변환, 음역(Transliteration)
* 문자 집합, 단어/문장 경계, 문자셋 감지 등 국제화 보조 기능

### 사용 목적

ICU4J는 Java 애플리케이션에서 Unicode 및 국제화 처리를 지원하기 위한 라이브러리이다. 일반적인 `java.text`, `java.util.Locale` 기능보다 넓은 범위의 Unicode 표준, CLDR 로케일 데이터, 다국어 정렬, 정규화, 음역, 문자셋 감지 기능을 제공한다.

`icu4j-53.1.jar`는 2014년에 배포된 구버전으로, eGovFrame 4.3 전환 시에는 현행 JDK 1.8 환경과 호환되는 상위 버전으로 업그레이드하는 것이 적절하다. 단, ICU4J는 버전 상승에 따라 Unicode/CLDR 데이터가 변경되므로 문자열 정렬, 날짜·숫자 포맷, 다국어 표시 결과가 일부 달라질 수 있다. 따라서 단순 JAR 교체 후 기능 정상 여부만 볼 것이 아니라, 업무 화면과 배치 결과의 표시값 회귀 검증이 필요하다.

### 대표 사용 예시

#### Unicode 정규화 예시

```java
import com.ibm.icu.text.Normalizer2;

public class IcuNormalizeSample {
    public String normalize(String value) {
        Normalizer2 normalizer = Normalizer2.getNFCInstance();
        return normalizer.normalize(value);
    }
}
```

#### 로케일 기반 문자열 정렬 예시

```java
import com.ibm.icu.text.Collator;
import com.ibm.icu.util.ULocale;

import java.util.Collections;
import java.util.List;

public class IcuCollationSample {
    public void sortKorean(List<String> names) {
        Collator collator = Collator.getInstance(ULocale.KOREAN);
        Collections.sort(names, collator);
    }
}
```

#### 음역 또는 문자 변환 예시

```java
import com.ibm.icu.text.Transliterator;

public class IcuTransliterationSample {
    public String toLatin(String value) {
        Transliterator transliterator = Transliterator.getInstance("Any-Latin");
        return transliterator.transliterate(value);
    }
}
```

## 3. AS-IS 사용 현황

### 프로젝트 내 사용 여부

* 확인 필요
* `Library Migration Matrix`의 27번 항목 기준으로는 `Utility / 국제화·문자 처리` 용도의 `버전업` 대상으로 식별되어 있다.
* 실제 사용 여부는 Java 소스의 `com.ibm.icu.*` import, Maven 의존성, WAS 공통 라이브러리 배치 여부를 함께 확인해야 한다.

### 확인 방법

#### Maven 의존성 확인

```bash
mvn dependency:tree -Dincludes=com.ibm.icu:icu4j
mvn dependency:tree | grep -i "icu4j\|com.ibm.icu"
```

#### Java 소스 사용 여부 확인

```bash
grep -R "com.ibm.icu" ./src ./config .
grep -R "Normalizer2\|Normalizer\|UCharacter\|ULocale" ./src ./config .
grep -R "com.ibm.icu.text.Collator\|Transliterator\|UnicodeSet" ./src ./config .
grep -R "CharsetDetector\|BreakIterator\|RuleBasedNumberFormat" ./src ./config .
grep -R "com.ibm.icu.text.DateFormat\|com.ibm.icu.text.NumberFormat" ./src ./config .
```

#### JAR 직접 반입 여부 확인

```bash
find . -iname "icu4j*.jar"
find . -iname "*icu*.jar"
```

#### 중복 버전 확인

```bash
find . -iname "icu4j*.jar" -exec ls -al {} \;
jar tf icu4j-53.1.jar | grep "com/ibm/icu" | head
```

### 예상 사용 위치

* 공통 문자열 유틸리티
* 이름, 주소, 기관명 등 다국어 문자열 정렬/검색 처리
* 파일 업로드, 전문, 연계 데이터의 문자셋 감지 또는 변환 처리
* 배치 데이터 정제 및 Unicode 정규화 처리
* 다국어 화면, 리포트, 엑셀/PDF 출력의 날짜·숫자 포맷 처리
* 외부 시스템 연계 전문의 인코딩 변환 또는 문자 검증 처리

## 4. eGovFrame 4.3 영향도 분석

### 전환 대상 여부

* 대상
* 조치 유형: 버전업
* 권장 전환 방향: JDK 1.8에서 동작 가능한 ICU4J 상위 버전으로 교체

### 전환 사유

`icu4j-53.1.jar`는 Maven Repository 기준 2014년 4월 1일 배포된 구버전이다. eGovFrame 4.3은 JDK 1.8, Servlet 3.1 환경을 요구하므로, 최신성을 확보하되 JDK 1.8 호환성이 확인되는 ICU4J 버전으로 제한하여 업그레이드해야 한다.

2026년 6월 기준 Maven Central의 최신 ICU4J는 78.3으로 확인되지만, ICU 78 계열은 Java 8에서 Java 11로 전환되는 흐름이 있으므로 eGovFrame 4.3의 JDK 1.8 운영환경에서는 무조건 최신 버전 적용이 아니라 JDK 1.8 호환 버전을 선택해야 한다. ICU 74.2는 공식 문서에서 Java 8~17 동작을 명시하고 있어 JDK 1.8 기반 사업에서는 1차 검증 기준 버전으로 적합하다. 다만 프로젝트 표준 저장소, 보안 점검, WAS/JDK 패치 수준에 따라 77.1 이하의 JDK 8 호환 후보도 함께 검토할 수 있다.

### 영향 범위

| 구분 | 영향 |
| --- | --- |
| 소스 | 낮음~중간. `com.ibm.icu.*` 패키지는 유지되므로 단순 사용은 수정이 적지만, deprecated API 또는 제거 API 사용 시 일부 수정 가능 |
| 설정파일 | 낮음. 일반적으로 Spring 설정 변경은 없으나, 공통 유틸 Bean 또는 문자셋/Locale 설정이 있으면 확인 필요 |
| pom.xml | 중간. 기존 직접 JAR 또는 구버전 의존성을 제거하고 `com.ibm.icu:icu4j` 상위 버전으로 명시 필요 |
| 운영환경 | 중간. Unicode/CLDR 데이터 변경으로 정렬순서, 날짜/숫자/통화 포맷, 시간대 표시, 문자 정규화 결과가 달라질 수 있어 회귀 테스트 필요 |

## 5. 전환 시 검토사항

### 변경 필요 사항

* `icu4j-53.1.jar` 직접 반입 제거
* Maven 또는 폐쇄망 Nexus 기준으로 `com.ibm.icu:icu4j` 의존성 관리
* eGovFrame 4.3의 JDK 1.8 환경에서 실행 가능한 ICU4J 버전 선정
* 소스 내 `com.ibm.icu.*` API 사용처 목록화
* 다음 업무 결과에 대한 회귀 테스트 수행
  * 한글/영문/특수문자/다국어 정렬 순서
  * Unicode 정규화 전후 문자열 비교
  * 날짜, 시간, 숫자, 통화 표시 형식
  * 파일/전문 인코딩 감지 및 변환 결과
  * 검색 키워드 생성, 초성/대소문자/전각·반각 처리 결과
* WAS 공통 라이브러리와 애플리케이션 WEB-INF/lib 중복 배치 여부 확인
* `icu4j-charset`, `icu4j-localespi` 등 보조 모듈 사용 여부 별도 확인

## 6. 버전 업그레이드 검토

### AS-IS 버전

```text
com.ibm.icu:icu4j:53.1
```

### TO-BE 버전

```text
1차 권장 검증 버전:
com.ibm.icu:icu4j:74.2

추가 검토 후보:
com.ibm.icu:icu4j:77.1 이하 JDK 8 호환 버전

비권장:
com.ibm.icu:icu4j:78.x 이상은 Java 11 요구 가능성이 있으므로 JDK 1.8 기반 eGovFrame 4.3 환경에서는 사전 검증 없이 적용하지 않음
```

### 주요 변경사항

* Unicode 및 CLDR 데이터 대폭 변경
* 로케일별 날짜, 시간, 숫자, 통화 포맷 데이터 변경 가능
* 다국어 문자열 정렬 기준 변경 가능
* Unicode 정규화, 문자 속성, 문자 경계 처리 결과 변경 가능
* 일부 deprecated API 존재 가능
* ICU 74 이후 Maven 프로젝트 구조 변경 사항 존재

### 소스 수정 여부

* 대부분 없음 또는 일부 수정 필요
* 단, 다음 조건에 해당하면 수정 필요 가능성이 높음
  * deprecated API 직접 사용
  * 내부 구현 클래스 `com.ibm.icu.impl.*` 직접 사용
  * ICU4J Locale Service Provider 또는 Charset SPI 보조 모듈 사용
  * 정렬/포맷 결과를 업무 로직의 비교 기준으로 직접 사용

## 7. 위험요소

### 보안

* `icu4j-53.1.jar`는 2014년 배포된 구버전이므로 보안 점검 및 오픈소스 취약점 점검 대상에 포함해야 한다.
* 알려진 취약점 여부는 사업 수행 시점의 SCA 도구 또는 CVE DB 기준으로 재확인해야 한다.
* 공공/폐쇄망 반입 시 POM, LICENSE, checksum, 공급 경로를 함께 보관해야 한다.

### 운영

* 날짜/숫자/통화 포맷이 기존 화면 또는 보고서 출력과 달라질 수 있다.
* 문자열 정렬 순서가 변경될 경우 목록 조회, 엑셀 출력, 검색 결과 순서에 영향이 발생할 수 있다.
* Unicode 정규화 방식 변경으로 문자열 비교, 중복 체크, 검색 키 생성 결과가 달라질 수 있다.
* JAR 크기가 큰 편이므로 중복 버전 존재 시 애플리케이션 패키징 크기와 클래스 로딩 충돌 위험이 있다.

### 호환성

| 항목 | 영향 |
| --- | --- |
| JDK 1.8 | 중간. eGovFrame 4.3 요구사항이 JDK 1.8이므로 ICU4J 버전 선택 시 가장 중요한 제약 조건 |
| Spring 5.3.37 | 낮음. ICU4J는 일반 유틸리티 라이브러리로 Spring과 직접 결합도는 낮음 |
| eGovFrame 4.3 | 낮음~중간. 프레임워크 직접 의존성보다는 업무 소스와 출력 결과 검증 영향이 큼 |
| WAS | 낮음~중간. WAS 공통 lib와 WEB-INF/lib 중복 배치 시 클래스 로딩 충돌 가능 |
| Unicode/CLDR | 중간. 버전업에 따른 데이터 변경으로 결과값 차이 발생 가능 |

### 비고

* `Library Migration Matrix`에 Git 병합 충돌 마커(`<<<<<<< HEAD`, `=======`, `>>>>>>> upstream/main`)가 남아 있으므로 최종 산출물 반영 전 Matrix 정리가 필요하다.
* 27번 항목의 기준은 `icu4j-53.1.jar`이며, 충돌 마커 아래쪽의 `httpclient-4.3.x.jar` 27번 항목과 혼동하지 않도록 번호 체계를 먼저 확정해야 한다.
* eGovFrame 4.3은 JDK 1.8 기반이므로 ICU4J 최신 버전을 무조건 적용하지 말고, JDK 1.8 호환성이 확인된 버전으로 제한해야 한다.
* ICU4J는 업무 기능을 직접 변경하는 라이브러리는 아니지만 결과 데이터가 화면, 파일, 보고서에 표시되므로 단순 컴파일 성공만으로 전환 완료 처리하면 안 된다.
* 실제 사용처가 없는 경우에는 버전업이 아니라 제거 가능성을 검토할 수 있다.
